# PACK 4 — Workforce & HR backend (`packages/api`) — business-logic review

Scope: worker / employee registry + lifecycle, personnel file (akta osobowe), leave, employee-time, ewidencja, payroll export, HRIS sync, HR dashboard. Method: read + grep + glob; traced tenant scope/IDOR, PII crypto, audit atomicity, tx boundary, Zod, flag re-assertion, statutory-math correctness. Source not modified.

## Systemic patterns

- **Audit not atomic with the mutation (recurring).** Multiple sensitive mutations `writeAuditLog` *after* the business write with no `$transaction`/`tx`: HRIS connect/disconnect/setMapping (`hris-sync-router.ts`) and lifecycle on/offboarding (`employee-lifecycle-router.ts:267`). Contrast the correct pattern in `leave.ts`, `ewidencja.ts`, `employee-time.ts`, `personnel-file/classify.ts`, and `hris-sync/apply-patch.ts`, which all pass `tx`. The `writeAuditLog` helper accepts `tx` precisely for this — the misses are omissions, not design.
- **Audit-coverage gap on the delete path.** The erasure router audits the *retained* (blocked) case but not the *erased* (data actually removed) case — the higher-risk branch is the unaudited one.
- **Inconsistent per-request flag re-assertion.** `assertWorkforceEnabled` is applied on most workforce procedures but missing on `classifyApprove`/`classifyReject`; the HR dashboard re-asserts only one of its two required flags. Registration-time gating is not a substitute for per-request re-assertion (the stated invariant).
- **Check-then-act on balances/registers.** Leave balance is validated at submit but mutated at approve with no re-check and no reservation (double-spend); ewidencja version bump is read-then-insert (bounded by a DB unique, so safe but surfaces raw P2002). Both are the read-then-write races the invariants call out.
- **PII crypto discipline is solid.** `employee-pii-crypto.ts`/`ssn-crypto.ts` use AES-256-GCM with per-class keys; national IDs are encrypted, `omit`-ted from create responses, exposed only via the audited `employeePii:read` `revealPii` path (org-scoped via `@unique(workerId)` + `organizationId` filter), and masked to last-4 in the payroll feed. HRIS field-partition (`field-partition.ts`) structurally prevents national-ID/financial write-back and two-way sync loops. No plaintext PII observed in logs or audit old/new values.
- **Tenant scoping / IDOR generally clean.** Every read/write restates `organizationId`; personnel-file section reads are permission-gated *before* the query (locked sections return no payload/count); HR dashboard doc-expiry filters by per-section grant; `buildPayrollFeed` and `applyPatchToWorker` inject `organizationId` as defense-in-depth over the tenant client.
