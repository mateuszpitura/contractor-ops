# Pack 3 — Classification / Contracts / Contractors / Compliance / Consent-GDPR: Business-Logic Review

Scope: `packages/api` routers (contractor, contract, classification*, ir35*, economicDependencyAlert, reassessmentTrigger, statusfeststellungsverfahren, complianceAdmin, consent, gdpr, classificationDashboard, classificationDocument) + services (contract-health, compliance-payment-gate, reassessment-trigger-scan, sanitize, rbac-recipients). Method: read + trace tenant scope / IDOR / audit / tx boundary / Zod / flag re-assertion / domain-rule correctness.

Tenant/IDOR baseline: `ctx.db` tenant extension (`packages/db/src/tenant.ts`) injects `organizationId` into `where` on all read + mutation-where operations, so `findFirst({ where: { id }})` is org-scoped. IDOR surface is therefore mostly covered; findings below focus on audit, tx atomicity, status-machine races, RBAC gating, and domain correctness.

## Systemic patterns

1. **Audit coverage is inconsistent across sensitive classification/DRV/SDS mutations.** In-tx audit is done well in `compliance-admin.ts`, `reassessment-trigger.ts`, `contract.ts` (`auditedMutation`), but is **absent** in classification `submit`, ir35-chain SDS delivery, and attestation upsert, and is **out-of-tx** in `statusfeststellungsverfahren.ts`. Every mutation named in the invariants (classification outcome/submit, SDS delivery, DRV clearance) should route audit through the same in-`$transaction` path.
2. **`writeAuditLog` without `tx` writes to the base `prisma` singleton, not the regional/tenant `ctx.db`.** This breaks both atomicity (audit can succeed/fail independently of the business row) and, for ME-region orgs, data residency. Prefer always passing `tx` inside a `$transaction`.
3. **Check-then-act on status machines.** `contract.transitionStatus`, `reassessment acknowledge/dismiss`, and `saveAnswer` all read state outside the mutating `where` guard. Move the guard into the write (`where:{ id, status/updatedAt: expected }`, assert affected count).
4. **RBAC gating is uneven within the classification namespace.** `statusfeststellungsverfahren`, `reassessmentTrigger`, `economicDependencyAlert`, and `classificationDashboard` layer `requirePermission(contractor:read|update)`; `ir35-chain` and `ir35-other-client-attestation` rely on `classificationProcedure` alone, letting read-only roles perform legally significant writes.
5. **Enum-casing drift between code and Prisma enums.** `classification-dashboard.activeAlertsByMarket` uses lowercase band values against the uppercase `EconomicDependencyBand`. Centralise enum constants (import from `@contractor-ops/db`) rather than hand-typing string literals in `where` clauses.

## Positives (not findings; context)
- `compliance-admin.ts` approve/reject flows: correct in-tx audit + document-ownership + PENDING_REVIEW guards.
- `classification-override` / `recreateComplianceAssessment`: server-derived verdict, audited, reason-required.

- CSV export: `escapeCsvField` neutralises formula-injection; signed URLs org-scoped, short TTL.
