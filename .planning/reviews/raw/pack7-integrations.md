# PACK 7 — Integrations, Outbound Webhooks, Public API, API keys, Gulf/Saudization, Import/Onboarding, Billing, Org/Settings/Portal

Production-grade business-logic review. Read from source (not memory). Scope = `packages/api` Pack 7 surfaces + the inbound webhook HTTP routes in `apps/api/src/routes/webhooks/` that back them.

## Summary verdict

The load-bearing security controls are **solid**: outbound SSRF guard (subscribe-time DNS + connect-time DNS-rebind re-validation, no redirect following), outbound HMAC signer (timestamp-bound, timing-safe, replay window), API-key HMAC hashing + fail-closed sandbox↔org isolation, inbound signature verification (multi-provider per-connection secret + Stripe `constructEvent`), Stripe idempotency (`StripeEvent.processedAt` inside a Serializable tx + transactional outbox), QStash signature verification on the drain, and per-org tier/quota gating on the public API. No auth-bypass, no plaintext secret storage, no SSRF bypass found.

The dominant real issue is **audit-not-in-transaction**: a correct `auditedMutation` / `writeAuditLog({tx})` pattern exists and is used in the newer surfaces (public-API writes, gulf overrides, deprovisioning override, org toggle, api-key `rotate`, free-zone `upsertAssignment`), but many other sensitive mutations call `writeAuditLog` *after* the business write with no shared `tx`. Since `writeAuditLog` throws on failure (audit-writer.ts:149), an audit failure leaves the mutation committed but unlogged (non-repudiation gap) and returns 500. Secondary issue: the CSV import `commit` re-accepts arbitrary client rows and skips the validate-step field validation.

## Systemic patterns

1. **Audit-not-in-transaction (dominant).** A correct in-tx audit path exists (`auditedMutation` in organization.ts; `writeAuditLog({tx})` in public-api/write-shared.ts, gulf override procedures, deprovisioning `overrideStepFailure`, free-zone `upsertAssignment`, api-key `rotate`). But ~13 other sensitive mutations (api-key create/update/revoke, all four webhook-subscription writes, jira `disconnect`, deprovisioning `enableProviderForOrg`, gulf `upsertConfig`/`upsertHeadcount`/`setSaudiAssignmentFields`, marketplace `update`, import `commit`) call `writeAuditLog` after the business write with no shared `tx`. `writeAuditLog` re-throws on failure (audit-writer.ts:149), so any audit failure = committed-but-unlogged mutation + 500. Recommend a lint/codemod to force every sensitive mutation through `auditedMutation`/`writeAuditLog({tx})`.

2. **Validate→commit trust gap in file import.** `import.parse`/`import.validate` do the real per-field validation, but `import.commit` re-accepts raw client `rows` and only re-checks `taxId`. The onboarding-import wizard (onboarding-import.ts) is better — it uses Zod input schemas and an optimistic-revision (`importJobsRevision`) tenant-isolated progress state. The CSV import path should adopt the same rigor.

3. **Inbound-webhook handler asymmetry.** Jira's inbound handler validates task transitions and scopes updates by `organizationId`; Linear's does neither. Both correctly `safeParse` payloads and rely on ingress-level signature verification (`apps/api/.../multi-provider.ts` per-connection secret, never trusting an `x-webhook-secret` header) + QStash-verified `_process` drain — that chain is sound.

3. **Positive controls confirmed (no action needed):** SSRF guard invoked at both subscribe (`webhook-subscription.ts:75/163`) and dispatch (`dispatcher.ts:249`) with a connect-time DNS-rebind `lookup` hook and no redirect following; HMAC signer timestamp-bound + `timingSafeEqual` + 5-min replay window; API keys HMAC-SHA256 hashed, plaintext shown once, sandbox↔org fail-closed (api-key-service.ts:161), suspended-org gate (api-key-auth.ts:60); Stripe `constructEvent` + 24h age gate + `StripeEvent.processedAt` idempotency in a Serializable tx + transactional outbox; webhook secret AES-256-GCM at rest; public-API writes audit-in-tx with `actorType:'API_KEY'` + captured IP/UA; portal auth is enumeration-safe, nonce-gated (IDOR), HMAC-session-bound, and region-aware on ownership checks.
