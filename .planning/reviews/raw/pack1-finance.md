## Systemic patterns

- **Audit-stream gaps on sensitive money mutations.** The payment-run family (`payment-run-ops`, `payment-export-router`, `payment-shared`, `ach-return.service`, `compliance-payment-gate`) writes `writeAuditLog` consistently, but several equally-sensitive finance mutations do **not**: BACS `generateExport`, LPC `waive`/`revokeWaiver`/`claim`, invoice `create`/`update`/`voidInvoice`/`toggleReverseCharge`. They rely on domain rows + `logger` instead of the shared append-only audit stream.
- **Non-atomic business-write + audit/side-write.** Multiple mutations perform the business write and its audit (or secondary write) as separate `await`s rather than one `$transaction`: `verifyBillingProfilePlaid`, `ingestAchReturnFile` (summary audit), `bacs.saveSubmitterConfig`, LPC `claim` (secondary invoice + claim). Any mid-sequence failure leaves a partial commit.
- **Check-then-act races outside a transaction** on uniqueness/status: invoice `create` duplicate hash, LPC `waive`/`claim` existence guards, `applySkontoToItem` eligibility. Most lack a DB unique constraint as a backstop.
- **Client-trusted state transitions with weak server-side enforcement.** `confirmStatementMatches`, `updateItemStatus`, and `markAllPaid` move money-bearing state (`PAID`/`COMPLETED`) based on client input without validating the payment-run state machine or re-verifying amounts.
- **Money-basis inconsistency for skonto.** The evaluate path uses `amountToPayMinor` while the apply path uses `totalMinor`, and apply clobbers the withholding-adjusted item amount — the two paths can disagree for reverse-charge/withholding/intake invoices.
- **FX staleness is unbounded** in `getRate`; combined with the 1-day-only ECB fallback, settlement and tax conversions can silently use very old rates instead of failing loudly (contra the documented "missing rate throws, never zeroes" invariant).
