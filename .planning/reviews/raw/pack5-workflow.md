# Pack 5 — Workflow / Approvals / Timesheets / Equipment — Business-Logic Review

Scope: `packages/api` workflow engine, approvals engine, timesheets, equipment/logistics.
Method: read actual source + wiki invariants. Focus: tenant scope/IDOR, audit-in-tx, transaction boundaries + approval races, per-resource RBAC, Zod, state-machine correctness.

## Findings

| `services/time-entry.ts:226,263,302,326` | BLOCKER | Timesheet `approveTimesheet` / `rejectTimesheet` / `bulkApproveTimesheets` / `bulkRejectTimesheets` write **NO audit log at all**. These records feed reconciliation → invoice → payment, so approvals of billable time have zero audit trail. | Add `writeAuditLog`/`writeAuditLogMany` inside the same write (wrap `updateMany` in `$transaction` and pass `tx`); record actor, timesheetId(s), old→new status, reason. |

## Systemic patterns

1. **Audit gaps on sensitive mutations are widespread.** The whole timesheet-approval surface (`time-entry.ts`) and the bulk approval path (`approval-queue.ts`), plus `delegate`/`requestClarification` and workflow `reassignTask`, perform state changes with no `writeAuditLog`. Single-action approval + equipment routers (`auditedMutation`) do it correctly — the gaps are the bulk/service-layer paths. Financial + billable-time actions are the most exposed.

2. **Check-then-act without an atomic guard on approval-style transitions.** `approve`/`reject`/`delegate` (approval), `approveReturnRequest`, and `equipment.assign`/`retire` read state, validate, then write unconditionally — no `updateMany({where:{…,status}})` guard or row lock. Timesheet service is the correct model (guarded `updateMany` with `count===0` check); the approvals engine and equipment routers should copy it. This is exactly the double-approve/race class the invariants call out.

3. **Two divergent run-completion code paths.** The guarded path (`unblockDependentsAndRecomputeRun` → `assertRunCompletable`, IP/credential gate) vs the ungated `equipment-workflow.recomputeWorkflowProgress`. The second silently completes runs and skips both the gate and audit — any alternate completion trigger must funnel through the single gated helper.

4. **External courier calls vs transaction boundary is inconsistent.** Router paths (`equipment-returns`, `equipment-couriers`) correctly call the carrier API before the tx; the workflow auto-shipment path (`equipment-workflow.autoCreateInPostReturnShipment`) does it *inside* the tx. Standardize on pre-tx external calls.

5. **State-machine enforcement is inconsistently applied.** DPD/UPS shipment creation filter by `EQUIPMENT_STATUS_TRANSITIONS`; InPost creation and several status flips do not. Centralize transition validation so no carrier/path can force an illegal equipment status.
