# PACK 2 — Tax & US/EU e-filing backend (packages/api) — business-logic review

Scope: `tax`, `taxForm`, `form1042s`, `form1099kTracker`, `tax1099`, `einvoice`, `leitwegId`, `peppol`, `ksef`, `zatca` routers + their services. Read against `init.ts` procedure ladder, `audit-writer`, tenant/region scoping, `$transaction` boundaries, Zod/`safeParse`, flag re-assertion, and tax/money/filing correctness.

Method: read + grep + schema cross-check (`packages/db/prisma/schema/tax.prisma`, `zatca.prisma`). No source modified.

## Findings

| File:Line | Severity | Problem | Recommended fix |
|---|---|---|---|
| `services/form-1099-nec.service.ts:504-595` | HIGH | `generateBatch` persists N ACTIVE rows + audit **outside any `$transaction`** and there is **no unique constraint** on `(organizationId, payerOrgId, recipientId, taxYear, status=ACTIVE)` (schema only has `supersededById @unique`). It does not supersede/dedupe before `create`. On a mid-loop throw (e.g. `aggregateBox1Async` throws on a missing FX rate for one recipient) the `catch` calls `clear(key)` while rows for earlier recipients are already committed; the retry re-creates them → **duplicate ACTIVE 1099s → `assembleIrisInput` emits duplicate payee B-records → double filing to the IRS**. Post-TTL (24h) re-run has the same effect. | Wrap the whole batch (per-recipient `create` + `writeAuditLog` + idempotency `complete`) in one `$transaction`, and supersede-before-insert per recipient (or add a partial unique index on ACTIVE rows). Do not `clear()` after rows are persisted without rolling them back. |
| `services/form-1042s.service.ts:601-712` | HIGH | Same defect as the 1099 engine: `generateBatch1042S` creates ACTIVE `Form1042S` rows + audit outside a transaction with no ACTIVE-row uniqueness and no supersede. Partial failure + `clear` + retry (or post-TTL re-run) → duplicate ACTIVE 1042-S rows → duplicate Pub 1187 recipient records on transmit. | Same as above: atomic transaction spanning create + audit + idempotency complete; supersede/dedupe per recipient. |
| `services/zatca-submission.ts:143-201` + `routers/compliance/zatca.ts:168-190` | HIGH | `submitToZatca` always calls `getNextChainEntry` (max icv+1) and `recordChainEntry` (create), but `ZatcaInvoiceChain.invoiceId` is `@unique`. ## Systemic patterns

- **Non-atomic "generate" writes (HIGH, recurring).** Both year-end engines (`form-1099-nec.service`, `form-1042s.service`) persist a batch of immutable filing rows + the audit row **outside a `$transaction`**, relying on an idempotency key that is `clear()`-ed on any failure. Combined with the absence of an ACTIVE-row uniqueness constraint (schema `tax.prisma` only guards `supersededById`), a partial failure or post-TTL re-run duplicates ACTIVE returns → duplicate government filings. The correction paths (`fileCorrection`/`fileCorrection1042S`) get this right (supersede-before-insert inside one tx); the generate paths should adopt the same discipline.
- **Missing audit on sensitive credential/certificate mutations.** The pack invariant "cert issuance … MUST writeAuditLog" is honored for the US 1099/1042-S filing surface (generate, correction, ack, FTIN reveal all audit) but **not** for WHT certificate issuance, ZATCA CSR/compliance/production-cert onboarding, or Peppol connect/disconnect. Audit coverage is inconsistent across the pack.
- **Global Prisma client in tenant/region-scoped services.** `tax-rate.service`, `treaty-rate.service`, `wht-certificate.service`, `zatca-onboarding`, and `zatca-submission` import the module-level `prisma`/`defaultPrisma` rather than `ctx.db`. For reference tables (`TaxRate`, `WithholdingTaxRate`, `ExchangeRate`) this is acceptable, but for **tenant data writes** (`WhtCertificate`, `IntegrationConnection`, `ZatcaInvoiceChain`) it bypasses RLS defense-in-depth and the EU/ME regional routing.
- **"Prefer specific over XX" lexicographic-sort trap.** `treaty-rate.service` documents and fixes it (two queries); `tax-rate.service.calculateWht` still ships the original buggy single-query `orderBy asc` form — the same anti-pattern surviving in one of two siblings.
- **Retry/idempotency asymmetry across the IRIS ledger.** The 1042-S transmit/ack path scopes every `IrisSubmission` lookup by `schemaVersionNum`; the 1099 path (`uploadAck`) does not, so the shared no-form-type-column ledger is only half-protected against cross-form ack contamination.
- **Strengths (no action):** flag re-assertion is consistent — every us-expansion procedure calls `assertUsExpansionEnabled` per request (`taxForm`, `form1042s`, `tax1099`, `form1099kTracker`), not just relying on `root.ts` registration. External/govt payload parsing is disciplined: `iris-ack-parser` is XXE-safe (DOCTYPE reject + fast-xml-parser + Zod `safeParse`, no `as` on the A2A shape); snapshot builders recursively strip full SSN/TIN/FTIN keys; the einvoice router, leitweg-id router/resolver, and the 1099-K tracker are cleanly tenant-scoped, transactional, and audited where they mutate. FTIN/TIN reveal is gated (`contractorPii:read`) and audited; PII masking (`tax-id-pii`, last-4 everywhere) is consistent.
