# Phase 56 — UI Design Contract

> Visual and interaction contract for Country Foundations & German i18n. Extends the Phase 47 Country Compliance section (UK + DE field groups), adds German as third platform locale, and ships jurisdiction-appropriate GDPR notices (web + PDF). Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope:** Contractor profile (UK/DE field groups), onboarding wizard (consent step extension), app chrome (language switcher + footer legal link), `/legal/privacy` MDX pages (UK/DE/EU), GDPR React-PDF template reuse.

## Typography

Existing site scale is reused. Only the roles actively consumed by this phase are declared below.

| Role                       | Size | Weight                      | Line Height | Notes                                                                 |
| -------------------------- | ---- | --------------------------- | ----------- | --------------------------------------------------------------------- |
| Body (default)             | 14px | 400 (Outfit regular)        | 1.5         | Form inputs, field labels body copy, MDX paragraph text               |
| Small / helper             | 12px | 400                         | 1.4         | Zod error messages, format-hint placeholders, footer legal link       |

| Display (page)             | 28px | 600 (Bricolage display)     | 1.15        | `/legal/privacy` page title ("Privacy Notice" / "Datenschutzerklärung") |

**Weights allowed in this phase:** exactly **two** — `400` (regular) and `600` (semibold). No italics introduced for DE. No weight 500.

**German expansion rule (LOAD-BEARING):**
- German copy averages **+30 %** character length vs English. All labels, CTAs, tab triggers, and Bundesland option rows MUST render on one line at **1024 px** (tablet breakpoint — our minimum per PROJECT.md) without truncation.
- Field labels wrap to two lines if needed; labels MUST NOT use `truncate`, `whitespace-nowrap`, or `overflow-hidden`.
- Buttons MUST use `whitespace-normal` only where the button contains sentence-length DE copy (onboarding CTAs). Icon-only buttons stay `whitespace-nowrap`.
- Card titles in DE use the same 18px/600/1.25 — no down-sizing.
- Bundesland dropdown row height: min 36px; uses `truncate` on the `<span>` but full name available via `title` attribute AND accessible name.

### 2. Bundesland dropdown UX (LOCKED DECISION)

- **Order:** alphabetical by **German name** (Baden-Württemberg → Thüringen). Default sort under any locale.
- **Option rendering:** `{name}` only. No population hints, no two-letter codes visible. Code is the underlying form value (e.g. `BW`).
- **Width:** matches parent `Input` width (full row).
- **Keyboard:** Radix `Select` default — `↑/↓` to navigate, `Enter` to select, type-ahead enabled.
- **Rationale:** Alphabetical is more discoverable for cross-state hires; population-sort would bury smaller states users actually need. No hint text (dropdown should be fast, not noisy).

### 3. Steuernummer input behaviour

- Disabled (greyed, not removed) until Bundesland chosen — placeholder shows: "Bitte Bundesland wählen" (DE) / "Select Bundesland first" (EN/PL).
- On Bundesland change, placeholder and masked format update to the selected state's example (e.g. NRW: `999/9999/9999`, Bayern: `999/999/99999`).
- The per-state regex + example map lives in `packages/validators/src/steuernummer-formats.ts` (new). Consumed both by the input and the Zod schema.
- Format hint appears as 12px muted text below the input: "Format: {example} · {Bundesland}".
- On blur, Zod validates with the Bundesland-specific regex. Error on fail; green checkmark on pass.

### 4. Handelsregister composite input

Three controls rendered in a single `<fieldset>` with one logical legend. MUST NOT be a single free-text input.

- **Layout (desktop, `≥640px`):** horizontal flex row — `[Court Select grows][HRB/HRA RadioGroup 88px][Number Input 120px]` with 8px gap.
- **Layout (mobile, `<640px`):** stacked vertical — each control full width, 8px gap.
- **Validation:** all three required together OR none at all. Partial completion is a form error: "All three parts are required …"
- **Example display:** helper row under the composite — "Example: Amtsgericht München · HRB · 123456".
- **Court Select:** shadcn `Select` listing the ~120 German registry courts. Source: `packages/validators/src/handelsregister-courts.ts` (new). Options sorted alphabetically by court name. Search/filter enabled (Radix Combobox pattern via shadcn `Command` inside `Popover`) because 120 options is too many for a plain select. Option label format: `Amtsgericht {city}`.
- **HRB/HRA RadioGroup:** two radios, horizontal on ≥640px, vertical on mobile. Labels: "HRB (Handelsregister B)" and "HRA (Handelsregister A)". Default: HRB (more common). Helper tooltip on HRA/HRB label explains the distinction briefly.
- **Number Input:** numeric mode, max 6–7 digits. `inputMode="numeric"` on mobile.

### 5. Accessibility grouping (Handelsregister)

One field, three controls — accessibility contract:

```html
<fieldset aria-labelledby="handelsregister-legend" aria-describedby="handelsregister-hint handelsregister-error">
  <legend id="handelsregister-legend">Handelsregisternummer</legend>
  <!-- court select, radio group, number input; each has its own visible sub-label but inherits the group legend -->
  <p id="handelsregister-hint">Example: Amtsgericht München · HRB · 123456</p>
  <p id="handelsregister-error" role="alert">{error or empty}</p>
</fieldset>
```

- Each sub-control has `aria-label` naming its role within the group ("Registry court", "Register type", "Registry number").
- Tab order: court → HRB/HRA → number → next field.
- Error message applies to the whole fieldset (single announced error), not to individual sub-controls.
- Radix `RadioGroup` manages arrow-key navigation within the radio pair.

### 6. Onboarding consent step extension

Existing Phase 51 flow for UAE/SA is untouched. For UK/DE orgs:

1. Consent step renders (was previously skipped — `isPdplJurisdiction` must expand to include UK + DE, or a new predicate `requiresPrivacyAcknowledgement(countryCode)` is introduced — planner decides).
2. Show `<PrivacyNoticeDisplay>` with UK/DE notice content (from `packages/validators/src/privacy-notices/{gb,de,eu}.ts` — new).
3. Render `<PrivacyNoticeAcknowledgement>` — unchecked by default, required, with:
   - Checkbox + linked label: "Ich habe die [Datenschutzerklärung]({url}) gelesen und verstanden." (link opens `/legal/privacy` in a new tab with `rel="noopener noreferrer"`).
4. "Weiter" button:
   - **State when unchecked:** `disabled` (visually muted, `aria-disabled="true"`, NOT removed from tab order — focusable with `tabindex="0"` so screen-reader users can understand why they can't proceed).
   - **State when checked:** teal primary, enabled.
   - **Click when enabled:** write `ConsentRecord` with `privacyNoticeVersion`, `privacyNoticeJurisdiction`, `acknowledgedAt` (server-side timestamp), `userAgent`, `ipHash` (same pattern as Phase 51 D-04).
5. Attempted submit without checkbox triggers `aria-live="polite"` error: "Please confirm you have read the privacy notice to continue." (DE: formal Sie equivalent.)

### 7. Footer legal link

- Persistent across the authenticated app shell (NOT on the marketing/login screens — those already have their own legal links).
- Placement: bottom of the main layout, 64px top padding above it on long pages, sticky on short pages (less common).
- Contents: `Privacy · Terms · © 2026 Contractor Ops · v{appVersion}`
- "Privacy" links to `/{locale}/legal/privacy`; routes to correct jurisdiction page based on `session.organization.countryCode` (UK → `/legal/privacy/uk`, DE → `/legal/privacy/de`, else → `/legal/privacy/eu`).
- Links: 12px, `--muted-foreground`; hover/focus `--primary` with underline.
- Keyboard: tab order places footer links after main content — reachable via sequential tabbing without a keyboard trap.
- Focus ring: 2px teal ring with 2px offset (shadcn default).
- Minimum tap target: 44×44 via vertical padding; text alignment centered within.

### 8. Language switcher — German entry

- Existing switcher (in top bar) gains a fourth option.
- **Option label rule:** render each locale in its **own native name**, in its own script:
  - `English` · `Polski` · `العربية` · `Deutsch`
- Current locale shows a check icon (Lucide `Check`) in front; selected row uses `bg-teal-50 text-teal-900`.
- Language labels use `lang="{locale}"` on the `<li>` (or `<option>`) so screen readers pronounce them with the right voice.

### 9. `/legal/privacy` MDX page contract

- Route: `apps/web/src/app/[locale]/legal/privacy/page.tsx` renders a jurisdiction picker **IF** org-less (unauthenticated visitor); `apps/web/src/app/[locale]/legal/privacy/[jurisdiction]/page.tsx` renders the specific notice.
- Authenticated users are auto-redirected to their org's jurisdiction.
- Heading hierarchy: exactly one `<h1>`; sections as `<h2>`; sub-sections as `<h3>`. No heading level skipped.
- Skip-link: first focusable element — `<a href="#main" class="sr-only focus:not-sr-only …">Skip to content</a>`. Visible on focus with teal background.
- TOC: `<nav aria-label="Table of contents">` left-column on ≥1024px, collapses to `<details>` on mobile.
- Scrollspy: current section `<h2>` is highlighted in TOC via `aria-current="location"`.
- Version banner: info-colored callout at top: "Version 1.0 · Effective 1 January 2026" — uses `--info` token, not accent.
- PDF download button: teal primary, positioned top-right of hero on desktop, full-width under title on mobile.
- `prose` Tailwind plugin NOT used (keeps control over heading tokens); MDX components map to our own heading set matching the Typography table above.

### 10. PDF / print view contract

PDF is a **distinct visual layer** from the screen MDX page. Re-uses Phase 51's React-PDF template pipeline and base styles.

- Paper: A4, 20mm margins all sides, single column.
- Font: React-PDF default Helvetica family (system-embedded) — **NOT** Outfit. Outfit is a web font and embedding it in React-PDF requires font-loading config; keeping Helvetica inherits Phase 51's tested setup.
- Colors: grayscale primarily; Deep Teal used only for the heading underline and first-page title accent. Destructive / warning colors are NOT used in the PDF.
- Header (every page): org logo (if present) + "Privacy Notice" / "Datenschutzerklärung" left-aligned; page X of Y right-aligned.
- Footer (every page): "Version 1.0 · Effective 2026-01-01 · {jurisdiction}" + generated timestamp.
- TOC: static numbered list on page 2; no hyperlinks required (PDF is read-only).
- Line height: 1.5 body, 1.25 headings.
- Font sizes: 10pt body, 12pt h3, 14pt h2, 18pt h1.
- File name: `privacy-notice-{jurisdiction}-{version}.pdf` (e.g. `privacy-notice-de-1.0.pdf`).
- Accessibility: PDF includes `<Document title="…" author="Contractor Ops" subject="Privacy Notice">` + language attribute matching locale.

## Accessibility Contract

WCAG **2.1 AA** baseline (PROJECT.md). Phase 56 specifics:

- **Color contrast:** existing tokens already meet AA; verified teal `oklch(0.44 0.145 178)` on `oklch(1 0 0)` = ~7.3:1 (AAA body). Field error destructive red on card white = ~5.2:1 (AA).
- **Focus visible:** every interactive element uses shadcn's 2px teal ring + 2px offset. No `outline: none` without replacement.
- **Keyboard:** no keyboard traps. Handelsregister composite navigable via Tab → RadioGroup arrow keys → Tab. Consent checkbox and Continue button reachable without mouse. Skip-link on `/legal/privacy` is the first focusable element.
- **Screen reader:** Bundesland + Steuernummer announce as related via `aria-describedby` — "Steuernummer format updates when you change Bundesland." Handelsregister fieldset announces as a single logical field (see §5 above).
- **Form errors:** `role="alert"` + `aria-live="assertive"` for validation errors triggered on submit; `role="status"` + `aria-live="polite"` for async validation results (blur).
- **Labels:** every input has a visible `<Label>` (shadcn). Placeholder is never the sole label.
- **Touch targets:** minimum 44×44px on tablet/mobile breakpoints (PROJECT.md desktop-first but approval flow works on mobile browser).
- **Language attributes:** `<html lang="de">` on DE pages; `lang="…"` attribute on any mixed-locale content (language switcher native names).
- **Long DE labels:** no truncation on labels or CTAs; line-wrapping permitted with 1.5 line-height.
- **Reduced motion:** dropdown open animations respect `prefers-reduced-motion` (Radix handles this).
- **Non-text content:** Lucide icons used for decoration in the Country Compliance card have `aria-hidden="true"`; the success checkmark on a validated field is role="img" with `aria-label="Valid format"`.
- **MDX privacy page:** proper heading hierarchy, anchored `<h2>` for each section, visible skip-link, TOC is `<nav>` with accessible name.
- **PDF accessibility:** React-PDF document has `title`, `author`, `subject`, and `lang` metadata; headings tagged semantically. (Full PDF/UA conformance is out of scope — Phase 51's baseline is inherited.)
