<scope_justification>
**Why Plan 06 is not split into 06a (FSM + finalize + read procedures) and 06b (send + webhook):**

1. **Transactional + FSM cohesion.** The lifecycle FSM (Task 1) is consumed by BOTH finalize (Task 1) and send + webhook (Task 2). Splitting would either duplicate FSM imports across two plans with divergent test harnesses, or force Plan 06b to re-read Plan 06a's SUMMARY to reconstruct transition semantics — both patterns known to drift.

2. **Single router file, single test harness.** `packages/api/src/routers/einvoice.ts` receives all 7 procedures in one diff; splitting would create a merge-conflict window where Plan 06a's procedures land first, Plan 06b appends, and any shared helper (e.g. `loadLifecycleScoped`) must be designed for two callers across two plans. Keeping it atomic lets Task 2 refactor shared helpers freely.

3. **End-to-end integration coverage in one run.** The send-path tests (Task 2) exercise the finalize-produced lifecycle rows (Task 1) and the webhook-driven state updates (Task 2) together. A single `pnpm test -- --run "einvoice"` invocation proves the full round-trip; splitting would require the send plan to mock or re-seed finalize output, losing real integration signal.

4. **Task budget still fits ~50% context.** Both tasks are `tdd="true"` with well-scoped test lists (Task 1: ≥11 tests across FSM + finalize; Task 2: ≥12 tests across router + webhook); the combined file set is 9 files modified — comparable to other Wave-3 plans in this phase.

5. **No checkpoint in this plan.** Fully autonomous; no user-interaction boundary forces a split.

If during execution the context budget turns out insufficient, the natural fallback is to stop after Task 1, commit, and resume Task 2 on a fresh context — but the plan is designed to complete in one pass.
</scope_justification>

Validation FSM:
  NOT_VALIDATED → VALID | INVALID | WARNINGS
  VALID → VALID | INVALID | WARNINGS            (re-validate may transition laterally)
