## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Outbound HTTP → Storecove API v2 | Carries org-scoped secrets + invoice payloads; existing HMAC + rate-limit layer is the gate. |
| Storecove response → adapter | Untrusted JSON; Zod validates shape. |
| Cache table → caller | `PeppolCapabilityCache` is read-mostly; invalid entries could cause wrongful "cannot send" UX. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-61-05-01 | Tampering | Untrusted Storecove response parsed without validation | mitigate | `storecoveDiscoveryResponseSchema` with `passthrough` + explicit normalizer; never trust documentTypes shape blindly. |
| T-61-05-02 | Information Disclosure | Cross-tenant capability cache leak | mitigate | Service always filters by `organizationId`; cache unique index is `(organizationId, schemeId, value)`; test #6 asserts isolation. |
| T-61-05-03 | Spoofing | Attacker forges a "capability" result to bypass send gate | mitigate | Cache is server-populated only (service owns writes); no client-write path; `forceRefresh` triggers live lookup. |
| T-61-05-04 | Denial of Service | Flooding `lookupCapabilities` with distinct schemeId/value to exhaust Storecove rate budget | mitigate | `GovApiRateLimiter` already enforces per-org bucket on Storecove calls; 6h cache absorbs repeat lookups; consider `forceRefresh` tRPC permission gate (admin-only) as a follow-up. |
