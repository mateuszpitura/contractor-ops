# Phase 58 — UI Design Contract

> Visual and interaction contract for the Classification Engine & Rule Sets phase. Covers the IR35 and Scheinselbständigkeit assessment wizards, blocking disclaimer modal, outcome pages (IR35 5-area cards + DRV traffic-light + weighted category bars), per-engagement Classification tile on the contractor profile, and per-contractor assessment history list. Inherits Phase 56 (`base-nova` / Precision Craft OKLCh) design system verbatim — **zero new tokens introduced**. Generated by gsd-ui-researcher, consumed by gsd-ui-checker, gsd-planner, and gsd-executor.

**Scope surfaces:** (1) classification entry point on contractor profile + engagement page, (2) multi-step wizard shell, (3) IR35 5-step wizard, (4) Scheinselbständigkeit 4-step wizard, (5) draft autosave indicator, (6) blocking disclaimer modal, (7) IR35 outcome page, (8) DRV outcome page, (9) classification assessment list per contractor, (10) i18n locked phrases + CI guard hookup.

**Explicitly out of scope (inherited from Phase 58 CONTEXT.md `<deferred>`):** regulatory-grade SDS PDF (Phase 59), DRV audit defense bundle (Phase 59), IR35 chain participant tracking (Phase 59), economic-dependency alert thresholds (Phase 60), reassessment triggers (Phase 60), Statusfeststellungsverfahren tracking (Phase 60), compliance health dashboard (Phase 60), bulk assessment runner, third-country rule sets.

### Accent reserved for (exclusive list — Deep Teal used on these elements only in Phase 58)

1. **Primary CTAs only:**
   - "Run classification assessment" on Classification profile tile and engagement page
   - "Next" button inside the wizard (when step is valid)
   - "Submit assessment" on the final wizard step
   - "I understand — view outcome" in the disclaimer modal
   - "Re-run assessment" on outcome page and assessment list
   - "Print / Export PDF" on outcome page (basic variant; full regulatory PDF is Phase 59)
2. Active/focus ring on form inputs and radio/likert option focus halo (`--ring` = teal) — Radix/shadcn default
3. Active step marker in the wizard step indicator (filled teal dot + 2px teal ring)
4. Progress bar fill (Radix `Progress` indicator uses primary by default)
5. Selected radio/likert option (filled teal dot + 2px teal ring; option card border shifts from `border-border` to `border-primary`)
6. Chevron icon when a help-text / case-law citation `Collapsible` is expanded (closed state uses `--muted-foreground`)
7. "View details" / "Expand breakdown" link hover/focus underline on outcome cards
8. Disclaimer modal "I understand" checkbox checked-state tick + 2px teal ring

**Accent NOT used for:**

- Area-card or DRV-category-card borders (use `--border`)
- Question prompt text or option labels
- Verdict pill fill or banner (those use the semantic triad — success / warning / destructive — for dispositive meaning; see "Color-independence rule" below)
- Step-indicator separator lines between steps
- Disclaimer modal body text or warning icon
- Autosave pill background (uses `--muted`)

### Semantic triad rules (IR35 verdict + DRV traffic-light)

**Color-independence rule (LOAD-BEARING — WCAG 1.4.1):**

Verdict and risk-level communication MUST NOT be colour-only. Every pill / banner / bar combines **colour + icon + text label**:

| State                  | Color token       | Lucide icon               | Text label (EN)   | Text label (DE)                     |
| ---------------------- | ----------------- | ------------------------- | ----------------- | ----------------------------------- |
| IR35 Outside           | `--success`       | `CircleCheck`             | "Outside IR35"    | "Außerhalb IR35"                    |
| IR35 Inside            | `--destructive`   | `CircleX`                 | "Inside IR35"     | "Innerhalb IR35"                    |
| IR35 Undetermined      | `--accent-warm`   | `CircleHelp`              | "Undetermined"    | "Unbestimmt"                        |
| DRV Green (low)        | `--success`       | `ShieldCheck`             | "Low risk"        | "Geringes Risiko"                   |
| DRV Amber (moderate)   | `--accent-warm`   | `ShieldAlert`             | "Moderate risk"   | "Mittleres Risiko"                  |
| DRV Red (high)         | `--destructive`   | `ShieldX`                 | "High risk"       | "Hohes Risiko"                      |
| Area-verdict leaning   | `--success` / `--destructive` at 50 % opacity | `TrendingUp` / `TrendingDown` | "Leaning outside" / "Leaning inside" | "Tendenz außerhalb" / "Tendenz innerhalb" |
| Area-verdict neutral   | `--muted-foreground` | `Minus`                   | "Neutral"         | "Neutral"                           |

**Accent (teal) is never used to communicate verdict — only for actions and navigation affordances.** This keeps the semantic load visually distinct from interactive load.

**Destructive (red) usage boundary for this phase:**

- Inside IR35 verdict (dispositive outcome) — banner + pill
- DRV red traffic-light (high risk outcome) — banner + category-bar fill when the category itself crosses the red threshold
- Inline validation error text + input border on optional free-text rationale fields
- **NOT** used on buttons. No destructive action exists in Phase 58 — "Re-run" creates a new row (append-only per D-04); "Cancel draft" discards but does not delete a completed assessment.

## Interaction & State Contracts

### 1. Assessment state machine (LOAD-BEARING)

The `ClassificationAssessment` row (D-04) walks this strictly-ordered state graph:

```
  (none)
    │  tRPC createDraft
    ▼
  draft  ───────────────┐
    │                   │  every answer change → saveAnswer (server-side upsert of `answers` JSONB)
    │                   ▼
    │              (still) draft
    │  tRPC submit
    ▼
  completed (outcome computed, stored, immutableAfter set)
    │  disclaimerAcknowledgedAt is NULL
    │  UI MUST render disclaimer modal before outcome
    │  tRPC acknowledgeDisclaimer
    ▼
  outcome-viewable (disclaimerAcknowledgedAt set)
```

**Transition rules enforced client-side:**

- `draft → completed` is a one-way transition. After submit, the wizard is no longer editable. Clicking "Re-run" creates a **new** draft row (append-only per D-04); it does NOT revive the old one.
- Between `completed` and `acknowledgedAt set`, the outcome page renders a loading skeleton and **force-opens** the disclaimer modal. Direct-linking to the outcome URL with `disclaimerAcknowledgedAt = null` MUST re-open the modal — the URL is shareable but the modal re-triggers.
- `acknowledgeDisclaimer` tRPC writes server-side timestamp; UI then closes the modal with a short exit animation (respects `prefers-reduced-motion`).

**Cannot-go-back rule:** after Submit, the Previous button is removed from the wizard (the wizard itself unmounts and redirects to outcome). Browser back button returns to the engagement page, NOT to the submitted wizard — the wizard route guards against state=completed by redirecting to the outcome URL.

### 2. Wizard step indicator + progress bar (accessibility LOAD-BEARING)

**Step indicator** (horizontal breadcrumb above the step card):

```html
<ol role="list" aria-label="Assessment steps" class="flex items-center gap-2">
  <li aria-current="step" class="flex items-center gap-2">
    <span class="step-dot filled">2</span>
    <span class="step-label">Control</span>
  </li>
  <!-- … -->
</ol>
```

- Visited steps: filled teal dot, label in `--foreground`.
- Current step: filled teal dot + 2px teal ring (`--ring`) + `aria-current="step"`, label in `--foreground` + weight 600.
- Future steps: outlined dot (teal border, transparent fill), label in `--muted-foreground`.
- On screens `< 640 px`: only the current step label is visible; others collapse to dots-only. Screen-reader text `<span class="sr-only">Step 2 of 5: Control</span>` is always present.

**Progress bar** (below step indicator, above step card):

```html
<div
  role="progressbar"
  aria-valuenow={currentStepIndex + (currentStepCompletion / 100)}
  aria-valuemin={0}
  aria-valuemax={totalSteps}
  aria-label={t('Classification.progress.ariaLabel', { current: currentStep, total: totalSteps })}
>
  <!-- Radix Progress primitive renders the fill -->
</div>
<p class="sr-only" aria-live="polite" aria-atomic="true">
  {t('Classification.progress.announce', { current: currentStep, total: totalSteps })}
  <!-- "Step 2 of 5: Control" -->
</p>
```

- `aria-valuenow` uses fractional completion (e.g. step 2 with 50 % of questions answered = 1.5) to communicate intra-step progress for screen readers.
- Live region announces step changes — NOT on every answer change (would be too chatty). Only when `currentStepIndex` changes.

### 4. Draft autosave contract (D-10)

- **Trigger:** Every answer change — `RadioGroup` select, `Input` blur, `Textarea` blur (debounced 500 ms for textarea only).
- **Transport:** tRPC `classification.saveAnswer({ assessmentId, questionId, answer })` — idempotent upsert into the `answers` JSONB.
- **UI feedback:** `<ClassificationAutosaveIndicator>` pill in the wizard shell:
  - Initial state: "Not saved yet" (muted)
  - Saving: "Saving…" + Lucide `Loader2` spinning icon (respects `prefers-reduced-motion`)

## Accessibility Contract

WCAG **2.1 AA** baseline (PROJECT.md). Phase 58 specifics:

- **Color contrast:** inherited semantic triad already meets AA — teal on white = 7.3:1, destructive red on white = 5.2:1, success green on white = 4.7:1, accent-warm amber on white = 4.8:1. Verdict pill uses white text on filled semantic backgrounds; contrast verified per token in globals.css.
- **Color independence:** verdict and risk communication ALWAYS colour + icon + text (table in §Color above). No user ever relies on colour alone to read the outcome.
- **Focus visible:** every interactive element uses shadcn 2 px teal ring + 2 px offset. Wizard Previous/Next, radio option rows, help-text chevrons, disclaimer modal controls, all outcome expand/collapse triggers. No `outline: none` without replacement.
- **Keyboard navigation:**
  - Wizard: Tab cycles through step content → Previous → Next (or Submit on final step) → autosave indicator → Cancel. No keyboard trap.
  - Radio groups: arrow keys navigate within a group, Tab moves out (Radix default).
  - Collapsibles: Enter/Space toggles expand/collapse.
  - Disclaimer modal: focus trapped inside; Tab cycles checkbox → Acknowledge later → Confirm (when enabled); Shift+Tab reverses.
- **Screen reader contract:**
  - Progress bar announces step changes via `aria-live="polite"` — only on step boundary, not per answer (avoids chatty mode).
  - Step indicator has `aria-current="step"` on the active `<li>`.
  - `aria-valuenow` on progress bar uses fractional completion; `aria-valuemax` = total steps.
  - Autosave pill uses `aria-live="polite"` + `aria-atomic="true"` so SR users hear "Saved 1 minute ago" transitions.
  - Legal-reference `Collapsible` reads as an expandable "Show legal reference" button with `aria-expanded` managed by Radix.
  - Outcome area cards: each card's verdict pill is given `role="status"` so the area verdict is announced when card comes into the reading order.
  - DRV category bars: each `<DrvCategoryBar>` announces "{category}: {weightedScore} of {max}. Risk level {green/amber/red}." via `aria-label` on the `role="img"` wrapper. The visual bar itself is `aria-hidden`.
- **Form errors:** `role="alert"` + `aria-live="assertive"` for submit-blocking errors (required criteria missing); `role="status"` + `aria-live="polite"` for async results (autosave saved, autosave failed).
- **Disclaimer modal:**
  - `role="alertdialog"` (Radix `AlertDialog`) — stronger than `dialog` for critical interruptions.
  - `aria-labelledby` → title id; `aria-describedby` → **body paragraph id covering the full disclaimer text** (not the checkbox label).
  - Initial focus on the checkbox (explicit, not auto-focus on Confirm).
  - Escape + overlay click DISABLED (see §6).
- **Labels:** every input has a visible `<Label>`. Placeholder is never the sole label (economic-dependency input has both "Billing ratio" label + "Percentage" `%` suffix).
- **Touch targets:** minimum 44 × 44 px on tablet/mobile breakpoints. Radio option card is 44 px tall minimum (`h-11` on touch, `h-9` on desktop).
- **Language attributes:** `<html lang="de">` on DE pages (inherited from Phase 56 next-intl). Locked German terminology inside an English-locale UI (rare — e.g. the word "Scheinselbständigkeit" cited in an EN tooltip) wrapped with `<span lang="de">Scheinselbständigkeit</span>` for correct pronunciation.
- **Bdi wrapping:** contractor/engagement names embedded in German sentences wrapped with `<Bdi>` (Phase 50 pattern). Example: DE outcome header — `<Bdi>{contractor.displayName}</Bdi> — Engagement {engagementId}`.
- **Long DE strings:** no truncation on step labels, question prompts, or verdict sub-lines. Line wrapping permitted at 1.5 line-height.
- **Reduced motion:** every transition (chevron rotate, progress bar fill, verdict banner entrance, modal exit) respects `prefers-reduced-motion: reduce`. Radix primitives handle this natively; Tailwind utilities use the `motion-reduce:` prefix.
- **Non-text content:** Lucide icons for decoration marked `aria-hidden="true"`. Verdict-pill icon has `role="img"` + `aria-label` matching the text label (redundant for colour-independence).
- **Assessment list table:** `<Table>` uses `<caption class="sr-only">{contractor name} classification assessments</caption>` so the table has an accessible name.
- **Print output:** page meta `<title>` includes `"{contractor} — {ruleSetVersion} — {verdict}"` so print driver uses it as filename fallback. No a11y regression — the print view is a visual transform; semantic HTML is preserved.

## Open Items (Claude's Discretion — locked for planner + executor)

These are explicitly permitted by CONTEXT.md `<decisions>` "Claude's Discretion" block. The UI researcher binds them here; the planner implements.

1. **Wizard primitive choice:** build a custom wizard shell from `Card` + `Progress` + step-indicator ordered list + Previous/Next buttons (NOT shadcn `Tabs` — Tabs implies equal-status sections; a wizard is strictly sequential). Rationale: clearer mental model + easier to enforce submit-once state machine.
2. **Server- vs client-rendered outcome:** **server-rendered** (Next.js App Router RSC). Rationale: the outcome page is an authenticated internal admin view; SEO doesn't apply, but SSR gives deterministic print output and avoids hydration flicker of the semantic-triad banner. Interactive bits (Collapsible expand, Print button) are `'use client'` leaves.
3. **Assessment list layout:** `<Table>` for ≥1024 px, vertical card list for < 1024 px. Rationale: multi-column density on desktop, touch-friendly on tablet.
4. **Print CTA label:** "Print / Save as PDF" (EN) — single button triggers `window.print()`; the browser print dialog offers native "Save as PDF" on macOS/Windows/Linux. Rationale: zero server-side PDF work in Phase 58; the regulatory-grade PDF is Phase 59.
5. **Progress bar fractional `aria-valuenow`:** yes, use step-index + intra-step completion fraction (e.g. step 2 of 5 with 3/6 answered = `aria-valuenow="1.5"`; `aria-valuemax="5"`). Rationale: screen-reader users deserve the same progress fidelity as sighted users seeing the bar.
6. **Likert 5-point vs Yes/No per IR35 question:** decided per-question inside the rule-set constant (`answerType` field). The UI researcher does NOT lock the inventory — the planner assigns per question with Steuerberater / UK tax adviser review (CONTEXT.md §Discretion). UI just renders whichever `answerType` the constant declares.
7. **DRV "Not applicable" scoring UX:** always shown as the first option in the `Score03Answer` group (score = 0), visually de-emphasised (`--muted-foreground` label colour) so it is present but not preselected. Rationale: D-11 requires 'Nicht anwendbar' as a valid answer without making it the path of least resistance.
8. **Disclaimer modal button order:** on LTR locales (EN/PL/DE): `[Acknowledge later] [I understand — view outcome]` — secondary/deferral-left, primary-right (shadcn convention). On RTL (not in scope for Phase 58 but for consistency with Phase 50): mirrored.
9. **Category bar threshold markers:** render as 1-px vertical lines in `--border` colour (not accent) at `0.3 × max` and `0.6 × max`. Label on hover via tooltip: "Grün-Schwellenwert / Amber-Schwellenwert". Rationale: informational chrome, not interactive — never accent.
10. **Outcome-page jump navigation:** at ≥1280 px a sticky right-rail TOC listing the 5 IR35 areas or 4 DRV categories, with scrollspy `aria-current="location"` on the current section. At < 1280 px the jump nav is absent. Rationale: progressive enhancement; the primary reading order flows top-to-bottom without requiring jump links on smaller screens.
11. **Relative-time library:** use `Intl.RelativeTimeFormat` (native, locale-aware). No third-party `date-fns`/`dayjs` dependency for this tiny slice. Rationale: zero-cost, universally supported.
