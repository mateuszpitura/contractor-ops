<objective>
Widen the shared `EInvoiceProfile.generate(invoice)` interface to `generate(invoice, opts?: unknown)` so each profile is free to narrow `opts` to its own domain-specific type at the implementation level, without breaking the shared contract.

Purpose: Phase 68 D-07 — Plan 02 (XRechnung) and Plan 04 (ZUGFeRD) both need to add a `skontoTerm?: SkontoTermInput | null` field to their per-profile options. Doing that on the shared interface directly would force every other profile (KSeF, ZATCA, Peppol-AE) to acknowledge a Skonto-shaped param it has no business with. Widening to `unknown` lets each profile narrow the param at its own boundary while keeping the shared interface jurisdiction-neutral.

Output: One-line type change in `packages/einvoice/src/types/profile.ts`. Zero runtime impact.
</objective>

<threat_model>
- THREAT: Widening `opts` to `unknown` lets a future profile accidentally accept a wrong-shaped opts object at runtime → MITIGATION: each profile (Plan 02, Plan 04, plus existing KSeF/ZATCA/Peppol-AE) narrows `opts` at its own implementation signature; TypeScript enforces the narrower type at every direct call site (not via the shared interface). Direct calls through the `EInvoiceProfile` reference would only see `unknown` and would be required to type-assert before use, which is the desired behavior — generic code working through the shared interface should not be Skonto-aware.
- THREAT: Some existing profile implementation accidentally typed `opts` as `void` and breaks → MITIGATION: per CONTEXT.md D-07, KSeF / ZATCA / Peppol-AE all currently accept `invoice` only (no `opts` arg). Adding an optional second param to the shared interface is structurally compatible with their existing 1-arg signatures (TypeScript treats `(invoice) => …` as assignable to `(invoice, opts?: unknown) => …`).
- THREAT: A future maintainer interprets `opts?: unknown` as "anything goes" and starts shoving runtime data through the shared interface → MITIGATION: Plan 02 sets the precedent (re-exports `XRechnungGenerateOptions` from the profile module) and the per-profile narrowed signature is the documented usage pattern. The shared interface is type-only; runtime callers always go through the profile-specific class.
</threat_model>
