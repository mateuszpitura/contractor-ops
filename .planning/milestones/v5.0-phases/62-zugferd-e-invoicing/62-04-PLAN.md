```ts
import type { PrismaClient, Contractor } from "@prisma/client";
import { logger } from "@contractor-ops/logger";

export type MatchReason = "VAT_ID" | "LEITWEG_ID" | "EXACT_NAME" | "FUZZY_NAME";

export type MatchCandidate = {
  contractorId: string;
  displayName: string;
  vatIdentifier: string | null;
  score: number; // higher is better
  reasons: ReadonlyArray<{ reason: MatchReason; detail?: string }>;
};

export type ExtractedSupplier = {
  supplierVatId?: string | null;
  supplierLeitwegId?: string | null;
  supplierName: string;
};

export async function rankIntakeCandidates(
  db: PrismaClient, // assumed already org-scoped via the project's Prisma extension
  orgId: string,
  extracted: ExtractedSupplier,
): Promise<MatchCandidate[]>;
```

**Strategy (deterministic, highest score first):**

1. If `extracted.supplierVatId`: query `db.contractor.findFirst({ where: { organizationId, vatIdentifier: { equals, mode: 'insensitive' } } })` → reason `VAT_ID`, score 100.
2. If `extracted.supplierLeitwegId`: query `db.leitwegId.findMany({ where: { organizationId, value: extracted.supplierLeitwegId }, include: { contractor: true } })`. For each distinct contractor: reason `LEITWEG_ID`, score 90.
3. Exact normalised-name match:
   - Normalise: `s.toLowerCase().trim().replace(/\s+/g, ' ').replace(/\b(GmbH|UG|AG|Ltd\.?|Limited|Inc\.?|KG|OHG|GbR|e\.V\.|SE)\b/gi, '').trim()`
   - `db.contractor.findMany({ where: { organizationId, name: {} } })` — load with the pre-filter optimisation below.
   - Keep candidates whose normalised name === normalised extracted name: reason `EXACT_NAME`, score 70.
4. Fuzzy: for each remaining contractor, compute Levenshtein distance on normalised names. Pre-filter first: only contractors whose normalised-name first-3-chars match extracted first-3-chars (case-insensitive); compute distance only on that subset. Keep entries with `distance <= 3`: reason `FUZZY_NAME` with `detail: "distance ${n}"`, score `50 - distance*5`.

Merge by `contractorId` — if a contractor matches multiple strategies, sum their scores and aggregate reasons. Sort desc by score. Truncate to top 5.

Implement a tiny Levenshtein function in a helper (≤ 40 LoC, Wagner–Fischer, no new dep).

Use `logger.child({ module: 'invoice-intake-matcher', orgId, vatId: extracted.supplierVatId ?? null })` for diagnostic logs — NEVER `console.*`. DO NOT log `supplierName` (potential PII).

Tests (`__tests__/invoice-intake-matcher.test.ts`) using an in-memory Prisma mock (`vi.fn()` stubs on `contractor.findMany`, `leitwegId.findMany`):
1. Given VAT-ID-exact + fuzzy-name match on the same contractor, score aggregates and reason includes both.
2. VAT-ID on contractor A + name-exact on contractor B → A ranks first (100 > 70).
3. No extracted VAT ID + no Leitweg → only name strategies run.
4. Fuzzy-only match with distance 3 → score 35; distance 4 excluded.
5. First-3-char pre-filter: fuzzy is NOT attempted against `"Zebra GmbH"` when extracted is `"Alpha GmbH"`.
6. German-suffix normalisation: `"Alpha GmbH"` and `"Alpha AG"` both normalise to `"alpha"` and collide → EXACT_NAME on both (score 70 each; neither outranks).
7. Returned list is capped at 5.

Create `packages/api/src/services/invoice-intake-service.ts`. Exports:

```ts
export type UploadInput = {
  orgId: string;
  userId: string;
  fileKind: "xml" | "pdf";
  fileBase64: string;
  mime: string;
  originalFilename: string;
};

export type UploadResult =
  | { kind: "CREATED"; intakeId: string; profileLevel: ZugferdConformanceLevel; validationStatus: InvoiceIntakeValidationStatus; warnings: string[] }
  | { kind: "DEDUP_RETURNED"; intakeId: string };

export async function uploadAndPersist(db: PrismaClient, input: UploadInput): Promise<UploadResult>;
export async function confirmMatch(db: PrismaClient, opts: { orgId: string; intakeId: string; contractorId: string; contractId?: string }): Promise<void>;
export async function acknowledgeValidation(db: PrismaClient, opts: { orgId: string; intakeId: string; userId: string }): Promise<void>;
export async function convertToInvoice(db: PrismaClient, opts: { orgId: string; intakeId: string; userId: string }): Promise<{ invoiceId: string }>;
export async function reject(db: PrismaClient, opts: { orgId: string; intakeId: string; userId: string; reason: string }): Promise<void>;
```

**`uploadAndPersist` flow** (strict order, typed errors at each gate):

1. **Size gate:** `Buffer.from(fileBase64, 'base64').length` must be ≤ 5 * 1024 * 1024. Else throw `{ code: 'FILE_TOO_LARGE' }`.
2. **MIME gate:** `mime` must be `application/xml` / `text/xml` for xml kind, or `application/pdf` for pdf kind. Else throw `{ code: 'UNSUPPORTED_MIME' }`.
3. **Hash:** `rawSha = createHash('sha256').update(bytes).digest('hex')`.
4. **Dedup pre-check:** `db.invoiceIntakeRequest.findUnique({ where: { organizationId_rawFileSha256: { organizationId: orgId, rawFileSha256: rawSha } } })`. If found, return `{ kind: 'DEDUP_RETURNED', intakeId }` immediately.
5. **Parse:** if `fileKind === 'pdf'`, call `parseZugferdPdf(bytes)`; else decode bytes to UTF-8 string (strip BOM) and call `parseXrechnungCii(xmlString)`. Both can throw: `ZUGFERD_NO_XML_ATTACHMENT`, `ZUGFERD_PDF_UNREADABLE`, `ZUGFERD_LEVEL_UNSUPPORTED`, `CII_PARSE_FAILED`. Re-throw typed.
6. **KoSIT validation** on the extracted XML: call `validateXrechnungCii(xmlString)` (re-export alias `validateZugferdEmbeddedXml`). If layer-1 (XSD) fails, throw `{ code: 'CII_XSD_INVALID', errors: firstFive }`. Layer-2/3 failures populate `validationStatus: 'INVALID' | 'WARNINGS'` — NOT thrown.
7. **R2 uploads** (via `packages/api/src/services/r2.ts`), in order:
   - raw file → `einvoice-intake/{orgId}/{sha[0:16]}.{pdf|xml}`
   - extracted xml (pdf uploads only; xml uploads share the raw key) → `einvoice-intake/{orgId}/{sha[0:16]}-extracted.xml`
   - validation report (HTML from KoSIT) → `einvoice-intake/{orgId}/{sha[0:16]}-{ruleSetVersion}-report.html`
8. **Persist** via `db.invoiceIntakeRequest.create({ data: {...} })`. On P2002 unique-violation on `(organizationId, rawFileSha256)` (race), re-fetch + return `DEDUP_RETURNED`.
9. **Status derivation:** validationStatus VALID → status `PARSED`. WARNINGS / INVALID → status `NEEDS_REVIEW`. EXTENDED level → push warning `LEVEL_EXTENDED_BEST_EFFORT` to row's `unmappedFieldsJson.warnings` and keep status `NEEDS_REVIEW`.
10. Return `{ kind: 'CREATED', intakeId, profileLevel, validationStatus, warnings }`.

**State-machine procedures:**

- `confirmMatch`: assert intake.organizationId === orgId; assert status IN ('PARSED', 'NEEDS_REVIEW'); update `matchedContractorId`, `matchedContractId`, status → `MATCHED`.
- `acknowledgeValidation`: assert org ownership; assert `validationStatus` IN ('WARNINGS', 'INVALID'); set `validationAcknowledgedAt = now()`, `validationAcknowledgedByUserId = userId`.
- `convertToInvoice`: assert status === `MATCHED`; assert `validationStatus === 'VALID'` OR `validationAcknowledgedAt !== null`; if already converted (convertedInvoiceId set), return that id (idempotent). Inside `db.$transaction`: create `Invoice` + `InvoiceLine`s from `parsedInvoiceJson` using the matched contractor/contract; update intake `status = 'CONVERTED'`, `convertedInvoiceId = newInvoice.id`. Return `{ invoiceId }`. Also write a `EInvoiceLifecycleEvent` of type `GENERATED` if applicable (or the new `ZUGFERD_GENERATED` only on outbound — so NOT here; here we link the source intake via a generic lifecycle entry if any).
- `reject`: assert status !== 'CONVERTED'; set `status = 'REJECTED'`, `rejectionReason = reason`. `reason.length >= 3` (Zod in router layer; service re-asserts defensively).

All mutations run inside `db.$transaction(async tx => ...)` when touching > 1 row.

Use `logger.child({ module: 'invoice-intake-service', intakeId, orgId })`. Never log raw file bytes or parsed invoice JSON.

Tests (`__tests__/invoice-intake-service.test.ts`) against a real ephemeral Prisma (the project's test pattern — check `packages/api/src/routers/__tests__/einvoice.test.ts` for the existing harness). 3. XML with XSD failure → throws `CII_XSD_INVALID`; `db.invoiceIntakeRequest.count` unchanged.
4. XML with schematron warnings → row created, status NEEDS_REVIEW, validationStatus WARNINGS.
5. PDF > 5 MB → throws `FILE_TOO_LARGE`; no R2 uploads performed.

7. `convertToInvoice` without `MATCHED` status throws `INVALID_STATE_TRANSITION`.
8. `convertToInvoice` twice (second call) returns the same invoiceId (idempotent).
9. `reject` on a CONVERTED row throws `INVALID_STATE_TRANSITION`.
10. Cross-org: `confirmMatch` with orgId mismatching intake.organizationId throws `NOT_FOUND`.
