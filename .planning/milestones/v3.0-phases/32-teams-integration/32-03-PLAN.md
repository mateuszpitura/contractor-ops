1. **TeamsBotHandler** (`packages/api/src/services/teams/teams-bot-handler.ts`):
   Extend `TeamsActivityHandler` from `botbuilder`:

   ```typescript
   import { TeamsActivityHandler, TurnContext, CardFactory } from "botbuilder";
   ```

   **Zod schemas for invoke payload validation** (per checker issue #6 -- external inputs must be validated):
   ```typescript
   import { z } from "zod";

   const approveInvokeSchema = z.object({
     action: z.literal("approve_invoice"),
     invoiceId: z.string().uuid(),
     flowId: z.string().uuid(),
   });

   const rejectInvokeSchema = z.object({
     action: z.literal("reject_invoice"),
     invoiceId: z.string().uuid(),
     flowId: z.string().uuid(),
   });

   const submitRejectionSchema = z.object({
     action: z.literal("submit_rejection"),
     invoiceId: z.string().uuid(),
     flowId: z.string().uuid(),
     comment: z.string().min(1, "Rejection comment is required"),
   });

   const taskModuleFetchSchema = z.object({
     invoiceId: z.string().uuid(),
     flowId: z.string().uuid(),
   });
   ```

   Implement:
   - **`onAdaptiveCardInvoke(context, invokeValue)`**: Per D-05, handle `approve_invoice` action:
     - Parse `invokeValue.action.data` through the appropriate Zod schema (`approveInvokeSchema` or `rejectInvokeSchema`). If validation fails, return `{ statusCode: 400, type: "application/vnd.microsoft.error", value: { code: "BadRequest", message: "Invalid payload" } }`.
     - Look up the acting user: get `aadObjectId` from `context.activity.from`, resolve to internal user via email match (query User by ExternalLink where externalType=TEAMS_USER and externalId=aadObjectId, or query by email from Graph API).
     - Call `advanceFlow({ flowId, userId: internalUserId, action: "APPROVE" })` from `approval-engine.ts`.
     - Return `{ statusCode: 200, type: "application/vnd.microsoft.card.adaptive", value: buildApprovalResultCard({ result: "approved", ... }) }`.
     - If flow already decided (ConflictError), return card showing "This invoice was already {status} by {name}."
     - If user lacks permission (ForbiddenError), return card showing "You don't have permission to approve this invoice."

   - **`handleTeamsTaskModuleFetch(context, taskModuleRequest)`**: Per D-05, return the reject modal card:
     - Validate `taskModuleRequest.data` with `taskModuleFetchSchema`. Return error task module on validation failure.
     - Extract `invoiceId`, `flowId` from validated data.
     - Return `{ task: { type: "continue", value: { title: "Reject Invoice", card: CardFactory.adaptiveCard(buildRejectModalCard(invoiceId, flowId)), width: "medium", height: "small" } } }`.

   - **`handleTeamsTaskModuleSubmit(context, taskModuleRequest)`**: Process rejection:
     - Validate `taskModuleRequest.data` with `submitRejectionSchema`. Return error on validation failure.
     - Extract `invoiceId`, `flowId`, `comment` from validated data.
     - Resolve user same as approve flow.
     - Call `advanceFlow({ flowId, userId: internalUserId, action: "REJECT", comment })`.
     - Update the original card via `context.updateActivity()` with `buildApprovalResultCard({ result: "rejected", ... })`.
     - Return `null` to close the modal.

   - **`onConversationUpdateActivity(context)`**: Per TEAM-06/D-13, capture ConversationReference:
     - `const ref = TurnContext.getConversationReference(context.activity)`.
     - Store in `IntegrationConnection.configJson` under `conversationReferences[aadObjectId]` key.
     - For team-scoped conversations, also store under `teamConversationReferences[teamId]`.
     - Use prisma to read existing configJson, merge the new reference, write back.

   Export helpers: `storeConversationReference(organizationId, ref)` and `getConversationReference(organizationId, aadObjectId)`.

2. **TeamsMessagingProvider** (`packages/api/src/services/messaging/teams-messaging-provider.ts`):
   Implements `MessagingProvider` with `platform = "teams"`:

   - `getUserId(organizationId, userId)`: Look up ExternalLink where `externalType = "TEAMS_USER"` and `entityId = userId`. Return `externalId` (aadObjectId). If not found, try to match by email via Graph API `getUserByEmail()`, create ExternalLink, return aadObjectId.

   - `sendApprovalCard(params)`: Get stored ConversationReference for recipientId. Use CloudAdapter `continueConversation()` to send the approval card via `CardFactory.adaptiveCard(buildApprovalCard(params))`. Store the sent activity ID in configJson for later card updates.

   - `sendChannelAlert(params)`: Get team ConversationReference for the target channel. Use `continueConversation()` to send `CardFactory.adaptiveCard(buildActivityAlertCard(params))`.

   - Helper for CloudAdapter singleton: create once with `ConfigurationBotFrameworkAuthentication({ MicrosoftAppId, MicrosoftAppPassword, MicrosoftAppType: "MultiTenant" })`.

Per D-11, create `docs/setup/teams-bot-setup.md` with step-by-step Azure Bot Service registration instructions:

1. **Prerequisites** section:
   - Azure account with active subscription
   - Admin access to Microsoft Teams tenant
   - Contractor Ops deployed with public HTTPS endpoint

2. **Step 1: Create Azure AD App Registration**:
   - Navigate to Azure Portal -> App registrations -> New registration
   - Name: "Contractor Ops Teams Bot"
   - Supported account types: "Accounts in any organizational directory" (multi-tenant) or "Single tenant" (see note about Jul 2025 deprecation)
   - Redirect URI: leave blank (bot handles its own auth)
   - Record the Application (client) ID -> this is `AZURE_BOT_APP_ID`
   - Go to Certificates & secrets -> New client secret -> Record value -> this is `AZURE_BOT_APP_SECRET`

3. **Step 2: Create Azure Bot Service**:
   - Navigate to Azure Portal -> Create a resource -> search "Azure Bot"
   - Bot handle: "contractor-ops-bot" (or custom name)
   - Pricing tier: Free (F0) for development, Standard (S1) for production
   - Link to the app registration from Step 1
   - Set messaging endpoint: `https://{your-domain}/api/teams/messages`

4. **Step 3: Enable Microsoft Teams Channel**:
   - In the Bot Service resource -> Channels -> Microsoft Teams
   - Accept the terms of service
   - Configure messaging features (enable bot)

5. **Step 4: Configure API Permissions** (for Graph API channel discovery):
   - In the App Registration -> API permissions
   - Add: Microsoft Graph -> Application permissions -> `Team.ReadBasic.All`, `Channel.ReadBasic.All`
   - Add: Microsoft Graph -> Delegated permissions -> `User.Read`, `offline_access`
   - Grant admin consent for the tenant

6. **Step 5: Set Environment Variables**:
   ```
   AZURE_BOT_APP_ID=<Application (client) ID from Step 1>
   AZURE_BOT_APP_SECRET=<Client secret value from Step 1>
   TEAMS_ENCRYPTION_KEY=<Generate with: openssl rand -hex 32>
   ```

7. **Step 6: Install Bot in Teams**:
   - In the Bot Service resource -> Channels -> Microsoft Teams -> Open in Teams
   - Or: create a Teams app manifest (manifest.json) and sideload in Teams Admin Center

8. **Troubleshooting** section:
   - "Bot not responding": Verify messaging endpoint URL is correct and HTTPS
   - "401 Unauthorized": Verify AZURE_BOT_APP_ID and AZURE_BOT_APP_SECRET match the app registration
   - "Cannot send proactive messages": Bot must be installed in user's personal scope first (triggers ConversationReference capture)
   - "Multi-tenant registration blocked": Microsoft deprecated new multi-tenant registrations after Jul 2025; use single-tenant configuration instead

Note: Do NOT create this as a markdown README. This is a setup guide requested by the user via D-11.
