<objective>
Fix the hardcoded `'readonly'` role in `retryFailedItem` so that retried invitations use the original role the user was imported with. Currently, when an import invitation fails and is retried, the retry always sends `'readonly'` regardless of the person's actual role — this silently downgrades users on retry.
</objective>

## Task 2: Use stored role in retryFailedItem instead of hardcoded 'readonly'

<read_first>
- packages/api/src/routers/onboarding-import.ts (lines 450-487 for retryFailedItem mutation)
</read_first>

<action>
In `packages/api/src/routers/onboarding-import.ts`, update the `retryFailedItem` mutation (around line 460-467).

Find the `createInvitation` call inside `retryFailedItem`:
```typescript
await auth.api.createInvitation({
  headers: ctx.headers,
  body: {
    email: input.email,
    role: "readonly", // Default role for retried items
    organizationId: ctx.organizationId,
  },
});
```

Replace with:
```typescript
const failedItem = job.failedItems[failedIndex]!;
await auth.api.createInvitation({
  headers: ctx.headers,
  body: {
    email: input.email,
    role: (failedItem.role || "readonly") as "admin" | "owner" | "finance_admin" | "ops_manager" | "team_manager" | "legal_compliance_viewer" | "it_admin" | "external_accountant" | "readonly",
    organizationId: ctx.organizationId,
  },
});
```

The `failedItem.role || "readonly"` fallback handles any pre-existing failed items that were stored before the role field was added.
</action>

<acceptance_criteria>

- `packages/api/src/routers/onboarding-import.ts` contains `failedItem.role` in the retryFailedItem mutation
- `packages/api/src/routers/onboarding-import.ts` contains `|| "readonly"` as a fallback for backwards compatibility
</acceptance_criteria>
