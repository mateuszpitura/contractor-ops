<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser UI → deprovisioning procedures | The trigger is a UI convenience; the server (81-02) is authoritative (RBAC + cooldown re-checked on submit). |
| UI permission gate | `usePermissions().can('idp', ['start_run'])` is advisory only — never the sole gate. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-81-05-01 | Elevation of Privilege | UI gate bypass | mitigate | UI `can('idp', ['start_run'])` is advisory; the server re-enforces `requirePermission({ idp: ['start_run'] })` on every call (81-02). A user editing the DOM still hits the server gate. |
| T-81-05-02 | Tampering | Client crafts arbitrary idempotencyKey/assignmentId | mitigate | assignmentId is resolved server-side (resolveAssignmentForContractor, org-scoped); the server re-runs the cooldown gate on submit (D-11) — the UI cannot bypass it. |
| T-81-05-03 | Information Disclosure | tRPC leaking into non-hook layer (data-layer drift) | mitigate | `check:web-vite-data-layer` gate in every task's verify; all tRPC confined to use-start-deprovisioning.ts. |
| T-81-05-04 | Denial of Service | Double-click double-start | mitigate | Deterministic per-assignment idempotencyKey (D-09) → server P2002 returns the existing run; UI shows "view run" once a run exists. |
| T-81-05-SC | Tampering | npm/pip/cargo installs | accept | No package installs (RESEARCH Package Legitimacy Audit — zero external packages; reuses existing @contractor-ops/ui primitives). |
</threat_model>
