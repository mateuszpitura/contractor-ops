<threat_model>
| ID | Threat | Mitigation |
|----|--------|-----------|
| T-77-02-01 | `tokens.delete` requires `admin.directory.user.security` scope the connection lacks → 403 mid-revoke | `classifyError` maps 403 forbidden → PERMANENT_FORBIDDEN; the GWS deprovision-scope const (77-01 / Phase 76) adds `.user.security`; missing-scope surfaces a lint:scopes / reconnect hint (D-07/D-08). Step fails cleanly, not silently. |
| T-77-02-02 | Partial revoke: some `tokens.delete` succeed, signOut fails → step reports SUCCEEDED but sessions remain | Both sub-actions are awaited; BOTH must return success for the step to be SUCCEEDED. Any sub-action failure → step FAILED with the failing sub-action's errorClass. Test asserts: signOut-fails ⇒ step FAILED even when all token deletes succeeded. |
| T-77-02-03 | SHA-256 hash embeds the access token or externalUserId (PII / secret leak in audit) | Hashes computed on canonicalised + PII-redacted payloads (token/authorization/externalUserId stripped before hashing — Phase 76 `saga-canonicalize` denylist). Test asserts a payload-with-token hashes identically to its redacted form. |
| T-77-02-04 | 429 on `tokens.delete` aborts the whole revoke instead of retrying | 429/503 → TRANSIENT_RATE_LIMIT → the method re-throws so the QStash step-runner retries the whole step with exp-backoff; idempotency (already-deleted token → 404 → treated as success) makes re-run safe. |
| T-77-02-05 | Concurrency on token deletes overwhelms the Admin SDK quota | `pLimit(5)` caps concurrent `tokens.delete`; `withResilience` honors Retry-After. |
| T-77-02-06 | `describeImpact` live calls run on EVERY saga UI peek, burning quota | describeImpact is cache-fronted by 77-04's preview service (5-min Upstash TTL); the adapter method itself does the live calls only on cache miss / force-refresh. |
</threat_model>
