Then edit the generated migration SQL file to contain:
```sql
-- Add generated tsvector column for full-text search
ALTER TABLE "Contractor" ADD COLUMN IF NOT EXISTS "search_vector" tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('simple', coalesce("legalName", '')), 'A') ||
    setweight(to_tsvector('simple', coalesce("displayName", '')), 'A') ||
    setweight(to_tsvector('simple', coalesce("taxId", '')), 'B') ||
    setweight(to_tsvector('simple', coalesce("email", '')), 'B')
  ) STORED;

-- GIN index for fast full-text search
CREATE INDEX IF NOT EXISTS "Contractor_search_vector_idx" ON "Contractor" USING GIN ("search_vector");
```

Add an `Unsupported("tsvector")` annotation comment in contractor.prisma above the model (as a comment only — do NOT add the column to the Prisma model since it's a generated column managed by PostgreSQL):
```prisma
// Note: "search_vector" tsvector column exists in PostgreSQL (managed by migration)
// but is excluded from Prisma schema as it is GENERATED ALWAYS AS and Prisma
// does not support generated columns. Query via $queryRaw.
```

**Step B: Create contractor tRPC router**

Create `packages/api/src/routers/contractor.ts` following the exact patterns from user.ts (imports, tenantProcedure, requirePermission). Import `prisma` from `@contractor-ops/db` and validators from `@contractor-ops/validators`.

Procedures to implement:

1. **`list`** — tenantProcedure + requirePermission({ contractor: ["read"] }) + input(contractorListSchema)
   - Build `where` clause from filters (status, lifecycleStage, ownerUserId, primaryTeamId, billingModel). Each filter is an `{ in: array }` condition.
   - If `search` provided and length >= 2: use `prisma.$queryRaw` to find matching contractor IDs via tsvector search. Construct search query by splitting on whitespace, appending `:*` for prefix matching, joining with ` & `. Include `WHERE "organizationId" = ${ctx.organizationId}` (CRITICAL for tenant isolation). Then add `{ id: { in: matchingIds } }` to the where clause.
   - If complianceHealth filter is provided, skip it in Prisma where — compute health after fetching and filter in memory (since health is computed, not stored).
   - Use `prisma.contractor.findMany` with `where`, `skip: (page-1)*pageSize`, `take: pageSize`, `orderBy: { [sortBy]: sortOrder }`, and `include`: owner (select id, name, image), primaryTeam (select id, name), billingProfiles (where isDefault, take 1, select billingModel-related fields), _count of complianceItems where status is MISSING or EXPIRED.
   - Use `prisma.contractor.count` with same where for total.
   - For each contractor, compute a simplified health badge: red if any MISSING/EXPIRED compliance items, yellow if any PENDING, green otherwise. This is the list-level health (not full computation).
   - If complianceHealth filter provided, post-filter the results by health value.
   - Return `{ items, total, page, pageSize }`.

2. **`getById`** — tenantProcedure + requirePermission({ contractor: ["read"] }) + input(z.object({ id: z.string() }))
   - Fetch contractor with all relations: owner, primaryTeam, primaryProject, defaultCostCenter, billingProfiles (ordered by isDefault desc), complianceItems (include requirementTemplateId), contracts (take 5, orderBy createdAt desc, select key fields), _count of workflowRuns, _count of invoices.
   - Compute full compliance health using the function from step C below.
   - Throw NOT_FOUND if contractor doesn't exist.
   - Return contractor with computed health.

3. **`create`** — tenantProcedure + requirePermission({ contractor: ["create"] }) + input(contractorCreateSchema)
   - Create Contractor with company and assignment fields. Set `organizationId: ctx.organizationId`, `lifecycleStage: "DRAFT"`, `status: "ACTIVE"`.
   - Create ContractorBillingProfile in the same transaction: `legalEntityName` from input.legalName, `preferredCurrency` from input.currency, `billingModel` stored in a field (use the contractor's `notes` field as a workaround, OR add a custom field — actually the billingModel is on the billing profile, but the schema doesn't have a billingModel field. Use `customFieldsJson` on the billing profile... wait, billing profile doesn't have that).

   Actually, looking at the Prisma schema: ContractorBillingProfile does NOT have a `billingModel` column. The billingModel lives conceptually at the contractor level but is not in the schema as a column either. Store it in `Contractor.customFieldsJson` as `{ billingModel: input.billingModel }`. Similarly for rateValueGrosze — store in customFieldsJson: `{ billingModel, rateValueGrosze }`.

   Create the billing profile with: legalEntityName, preferredCurrency, bankAccountMasked (last 4 digits of IBAN if provided), bankAccountEncrypted (for now, store the full IBAN — proper encryption is a future concern), paymentTermsDays, countryCode, validFrom: new Date(), isDefault: true.
   - Return the created contractor.

4. **`update`** — tenantProcedure + requirePermission({ contractor: ["update"] }) + input(contractorUpdateSchema.extend({ id: z.string() }))
   - Update Contractor fields. If billing fields changed, update default billing profile.
   - Return updated contractor.

5. **`updateLifecycleStage`** — tenantProcedure + requirePermission({ contractor: ["update"] }) + input(contractorLifecycleTransitionSchema)
   - Validate transition is legal. Legal transitions:
     - DRAFT -> ONBOARDING, ACTIVE
     - ONBOARDING -> ACTIVE, ENDED
     - ACTIVE -> OFFBOARDING, ENDED
     - OFFBOARDING -> ENDED
     - ENDED -> ARCHIVED (via separate archive action)
   - Throw BAD_REQUEST if transition is invalid.
   - Update lifecycleStage. If stage is ENDED, also set status to INACTIVE. If stage is ACTIVE and status is INACTIVE, set status to ACTIVE.
   - Return updated contractor.

6. **`archive`** — tenantProcedure + requirePermission({ contractor: ["delete"] }) + input(z.object({ id: z.string() }))

**Step C: Compliance health computation helper**

Create a helper function `computeComplianceHealth` inside the contractor router file (or as a separate function at the top):

```typescript
type HealthFactor = {
  key: "documents" | "contract" | "tasks" | "invoices";
  status: "green" | "yellow" | "red";
  label: string;
  detail?: string;
};

function computeComplianceHealth(params: {
  complianceItems: Array<{ status: string; expiresAt: Date | null }>;
  activeContractCount: number;
  expiringContractCount: number;
  overdueTaskCount: number;
  unpaidInvoiceCount: number;
}): { overall: "green" | "yellow" | "red"; factors: HealthFactor[] }
```

Logic:
- Documents: red if any MISSING items, yellow if any EXPIRED or PENDING, green otherwise.
- Contract: red if no active contract, yellow if any expiring (end date within 30 days), green if active.
- Tasks: red if overdueTaskCount > 0, green otherwise. (Tasks don't exist yet in Phase 2, so default green.)
- Invoices: red if unpaidInvoiceCount > 0, green otherwise. (Invoices don't exist yet, so default green.)
- Overall: red if ANY factor red, yellow if ANY yellow, green otherwise.
