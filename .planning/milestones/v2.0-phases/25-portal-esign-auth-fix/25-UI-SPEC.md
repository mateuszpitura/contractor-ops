## Phase 25 Scope: UI Changes

Phase 25 is a backend auth fix with minimal frontend changes. No new visual components are introduced.

### Modified Components

| Component | File | Change | Visual Impact |
|-----------|------|--------|---------------|
| EmbeddedSigningModal | `apps/web/src/components/contracts/contract-detail/embedded-signing-modal.tsx` | Add `usePortalAuth?: boolean` prop; conditionally call `trpc.esign.getPortalSigningUrl` instead of `trpc.esign.getSigningUrl` | None -- identical rendering, different auth path |
| PortalPendingSignatures | `apps/web/src/components/portal/portal-pending-signatures.tsx` | Pass `usePortalAuth` to EmbeddedSigningModal | None -- passes prop to child |

### Interaction Contract

| Interaction | Behavior | Error State |
|-------------|----------|-------------|
| Portal contractor clicks "Sign Now" | EmbeddedSigningModal opens, calls `getPortalSigningUrl` via portalProcedure, iframe loads signing URL | FORBIDDEN error if contractor is not a recipient of the envelope; modal shows existing error state ("Unable to load signing session. Please try again later.") with "Return to Contract" button |
| Signing complete (postMessage) | Modal closes, success toast ("Document signed successfully"), pending list refetches | No change from Phase 15 behavior |
| Autenti redirect fallback | Centered card with "Continue to Autenti" button opens provider in new tab | No change from Phase 15 behavior |

### States Unchanged

All visual states (loading skeleton, iframe embed, redirect fallback, error, signing complete toast) remain identical to Phase 15 implementation. The only difference is the tRPC procedure called, which is transparent to the user.
