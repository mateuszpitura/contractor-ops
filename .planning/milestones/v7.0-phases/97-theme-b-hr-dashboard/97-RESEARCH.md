# Phase 97: Theme B — HR Dashboard - Research

**Researched:** 2026-07-05
**Domain:** A STAFF-side HR command surface that aggregates the shipped Theme-B employee data (registry P90, personnel file P91, leave/employee-time P92, on/offboarding P93) plus the v6.0 F1 compliance-document engine and F3 Gulf polish into five read-only dashboard widgets: headcount breakdown (HR-DASH-01), vacation-utilization (HR-DASH-02), document-expiry (HR-DASH-03), probation-end watchlist (HR-DASH-04), and a Saudization/Emiratisation rollup (HR-DASH-05). NOT the employee/manager self-service portal (that is P96). Mostly read-only `groupBy`/rollup aggregation over existing rows, gated by HR-role RBAC + a dedicated `module.hr-dashboard` kill switch on top of `module.workforce-employees`.
**Confidence:** HIGH (every reuse seam read at source at current HEAD; five CONTEXT/decision assumptions corrected against live code — see Corrections; the RBAC gate + per-section grain verified in `packages/auth`).

## Summary

Phase 97 is a **read-and-aggregate** phase. The staff dashboard aggregation idiom (`packages/api/src/routers/core/dashboard.ts` — `rateLimitedDashboardProcedure` + `$queryRaw` FILTER aggregates + `cachedSingleflight` + `readReplica`), the per-org report rate-limit (`reportRateLimitMiddleware`, 30/min/org), the `groupBy` idiom (`invoice-crud.ts:509`), the leave-balance materialized cache (`LeaveBalance` — `entitledMinutes`/`usedMinutes`/`carryoverMinutes` per worker-year), the v6.0 F1 expiry date math (`@contractor-ops/compliance-policy` — `daysUntilExpiryInTz`/`isExpired`), the F3 Saudization service (`saudization-dashboard.ts` — manual-headcount nationalisation rate), the HR-role RBAC model (`packages/auth/src/{roles,permissions}.ts`), the per-section personnel-file grain (`employeeFileA..D`), the KPI-card idiom (`components/dashboard/kpi-cards.tsx`), and the reports widget/hook/page pattern (`pages/dashboard/reports.tsx`) all exist and were read at source. The new `hrDashboard.*` procedures wrap these under an HR-role gate scoped to `ctx.organizationId`.

The one thing this research corrects hard: **three of the four data surfaces the widgets need are NOT present in the schema**, despite CONTEXT D-01 asserting "P90 promoted department / employment-type / contract-end / Saudization columns for exactly this dashboard." At HEAD, `EmployeeProfile` promoted **only** `countryCode`, `employmentStatus`, `saudizationCategory`, `etat`, `terminatedAt` (P90-CONTEXT D-01 literally says "at minimum Saudization category, employment fraction (etat), and employment status"). There is **no** `department`, **no** `employmentType` (contract type), **no** scheduled `contractEndDate`, and — across the entire tree — **no probation field at all** (`grep -rniE probation` over `packages/` + `apps/` returns zero hits). And `PersonnelFileDocument` carries **no** `expiresAt` and **no** document category, while `compliance-reminder-scan` is `ContractorComplianceItem`-only. So HR-DASH-01 (by department / employment-type / contract-end), HR-DASH-03 (employee document-expiry), and HR-DASH-04 (probation watchlist) have no data source to aggregate. This research resolves that with a **thin, additive Wave-1 data-layer plan** (six nullable indexed columns + two enums + the flag) mirroring P90's promoted-column pattern exactly — not a rebuild, a prerequisite. Everything downstream is read aggregation, UI, and i18n.

Reuse seams verified at source:

- **The staff dashboard aggregation idiom.** `rateLimitedDashboardProcedure = tenantProcedure.use(requirePermission({ report: ['read'] })).use(reportRateLimitMiddleware)`; `kpis` runs `fetchKpis` inside `cachedSingleflight(CacheKeys.dashboardKpis(orgId), CacheTTL.DASHBOARD_KPIS_BURST, () => readReplica(region, db => …))` with `$queryRaw` `COUNT(*) FILTER (WHERE …)` single-scan aggregates `[VERIFIED: packages/api/src/routers/core/dashboard.ts:34-90,327-339]`. **HR-DASH reuses the shape but NOT the `report:read` gate (Correction C5).**
- **Per-org report/dashboard rate-limit.** `reportRateLimitMiddleware` caps `report:${orgId}` at 30/min (Upstash sliding window, in-memory fallback, prod fail-closed) — "apply downstream of `tenantProcedure` so `ctx.organizationId` is guaranteed" `[VERIFIED: packages/api/src/middleware/report-rate-limit.ts:123-181]`. HR-DASH aggregates share this budget.
- **The `groupBy` idiom.** `ctx.db.invoice.groupBy({ by: ['status'], where: { organizationId, deletedAt: null }, _count: { id: true } })` `[VERIFIED: packages/api/src/routers/finance/invoice-crud.ts:509-533]`. HR-DASH-01 headcount is `employeeProfile.groupBy({ by: ['department' | 'countryCode' | 'employmentType' | 'employmentStatus'], where: { organizationId }, _count })`.
- **The leave-balance materialized cache.** `LeaveBalance` holds `entitledMinutes`, `usedMinutes`, `carryoverMinutes` per `(workerId, leaveTypeId, year)`, refreshed in-tx with each ledger insert (`recomputedAt`) `[VERIFIED: packages/db/prisma/schema/leave.prisma:105-123]`. HR-DASH-02 **reads this cache directly** (taken vs entitled) — it does NOT re-sum the ledger. The entitlement engine (`resolveEntitlementMinutes({ jurisdiction, leaveKind, tenureYears, etat })`) already scales by `etat` `[VERIFIED: packages/api/src/services/leave-balance.ts:45-65]`.
- **The v6.0 F1 expiry engine is PURE date math in `@contractor-ops/compliance-policy`.** `daysUntilExpiryInTz(expiresAt, tz, now)` and `isExpired(expiresAt, tz, now)` compute TZ-correct expiry boundaries; per-jurisdiction rules live in `policies/{de,ksa,pl,uae,uk,us}.ts` `[VERIFIED: packages/compliance-policy/src/expiry.ts:27-58]`. **This is the composable asset for HR-DASH-03**, not `compliance-reminder-scan` (Correction C4).
- **`compliance-reminder-scan` is Contractor-only.** It scans `ContractorComplianceItem` + `ContractorComplianceReminderState`, keyed on `contractorId`/`contractor.displayName`, with a two-pass digest throttle `[VERIFIED: packages/api/src/services/compliance-reminder-scan.ts:97-142]`. It has **no** employee/personnel/akta path — so HR-DASH-03's *employee* document-expiry cannot reuse it as a data source; it reuses the pure `compliance-policy` math over employee doc rows (C4). If a proactive reminder cron is ever wired (CONTEXT D-03, planner's discretion), the digest throttle here is the thing to reuse — but v7.0 ships read-only widgets.
- **The F3 Saudization service is manual-headcount-only, by design.** `computeSaudizationDashboard` derives `nationalisationRate` EXCLUSIVELY from the manual `SaudiHeadcountInput` (Nitaqat counts the whole workforce; the platform sees contractors only) and reads `band` read-through from `SaudizationConfig` — "NEVER recomputed or inferred from the rate. No band-derivation code path exists here by design" `[VERIFIED: packages/api/src/services/saudization-dashboard.ts:1-19,99-161]`. **HR-DASH-05's Emiratisation extension MUST preserve this posture** (Correction C6).
- **The HR-role RBAC model is shipped and precise.** `packages/auth/src/roles.ts` defines the four worker-model HR roles — `hr_admin`, `hr_manager`, `payroll_officer`, `leave_approver` — each granting the `employee` resource (plus per-section `employeeFileA..D`); `owner` = `allPermissions` `[VERIFIED: packages/auth/src/roles.ts:60,198-243]`. Critically, the base `employee` resource grant appears in **only** those four HR roles + `owner` — `admin`, `finance_admin`, `ops_manager`, `team_manager`, `legal_compliance_viewer`, `it_admin`, `external_accountant`, `readonly` do **not** hold it `[VERIFIED: grep 'employee:' packages/auth/src/roles.ts → lines 199,216,230,239 only, + owner allPermissions]`. So `requirePermission({ employee: ['read'] })` resolves to exactly **owner + the four HR roles** — the correct HR-Dashboard gate (Correction C5).
- **The per-section personnel-file grain is a first-class RBAC surface.** Sections A (master/leave), B (discipline), C (pay), D (other) are each their own resource (`employeeFileA..D`), "decided section-by-section — a payroll role can reach section C without seeing section B" `[VERIFIED: packages/auth/src/permissions.ts:55-63]`. Grants differ by role: `hr_admin` reads all four; `hr_manager` reads A/B/C/D (C read-only); `payroll_officer` reads **only** C; `leave_approver` reads **only** A `[VERIFIED: roles.ts:205-242]`. `PersonnelFileDocument.section` links each doc into exactly one section `[VERIFIED: packages/db/prisma/schema/personnel.prisma:72-74]`. The HR-DASH-03 document-expiry widget MUST filter its per-document rows by the caller's section grants (D-04, the load-bearing authz reuse).
- **The workforce routers dark-mount under `module.workforce-employees`.** `workforceRouters = { employeeRegistry, personnelFile, leave, employeeTime, ewidencja, … }`; `conditionalWorkforceRouters = isWorkforceRegistered() ? workforceRouters : {}`; spread into `appRouter` `[VERIFIED: packages/api/src/root.ts:186-198,264]`. `hrDashboard` mounts inside `workforceRouters` (dark with the rest of Theme B) AND re-asserts a new `module.hr-dashboard` per request.
- **The web-vite dashboard/report UI pattern.** `pages/dashboard/reports.tsx` is a thin `Suspense` → `ReportsPageContent` that composes widget wrappers, each calling a `use-*-report.ts` hook (the sole tRPC boundary) `[VERIFIED: apps/web-vite/src/pages/dashboard/reports.tsx:22-106]`; the KPI-card primitive is `components/dashboard/kpi-cards.tsx` / `compliance/dashboard/compliance-kpi-cards.tsx` (`KpiCard`) `[VERIFIED]`. Dashboard routes register in `router/dashboard-routes.tsx`; nav in `lib/nav-items.tsx` + `lib/navigation.ts` + `i18n/navigation.tsx` `[VERIFIED]`. The HR dashboard is a new `dashboard/hr` route + widget/hook set mirroring this, NOT the portal.
- **`EmploymentStatus` ≠ employment type.** The promoted `employmentStatus` enum is `ACTIVE | ON_LEAVE | SUSPENDED | TERMINATED` `[VERIFIED: packages/db/prisma/schema/employee.prisma:62-67]` — a lifecycle *status* axis, not the *contract type* (full-time / part-time / fixed-term) HR-DASH-01 breaks down by. These are two distinct facets (C1).

**The single most important finding — the widgets' data surfaces are only partially present, and CONTEXT overstates what P90 promoted.** Headcount-by-department, headcount-by-employment-type, headcount-by-contract-end, probation-watchlist, and employee-document-expiry all reference columns that do not exist at HEAD. The honest, minimal fix is a thin additive Wave-1 migration (six nullable, indexed columns + two enums, mirroring the exact P90 promoted-column + `__`-prefixed-migration pattern) so the downstream aggregation is real `groupBy`/date-window work, not a JSON scan or a fabricated column. This is precisely parallel to how Phase 96 found the `managerWorkerId` reporting-line edge missing (its C2) and added it in 96-01 before building on it.

**Primary recommendation:** (Wave 1) the additive data-layer + `module.hr-dashboard` flag foundation; (Wave 2) the RED net — HR-role RBAC fence (only `employee:read` passes; non-HR roles blocked), org-scoping (cross-org groupBy reads nothing), per-section doc-expiry grain (payroll_officer sees only section-C expiry, not B), dark-flag gating, and aggregation-correctness fixtures (headcount sums to total; utilization taken/entitled; probation 14/7/0 window boundaries); (Wave 3) the read/aggregate `hrDashboard` procedures that flip the RED net GREEN — headcount+utilization, doc-expiry+probation, Emiratisation; (Wave 4) the web-vite HR dashboard UI (route + hooks + wired sections + KPI/widget views + per-widget empty/degraded states + nav); (Wave 5) 5-locale i18n parity + docs-follow-code + graph/BM25.

## The Load-Bearing Decision — the HR-Dashboard RBAC gate

The dashboard reads employee PII-adjacent, section-sensitive data. The gate is the crux (D-04). Three options:

### Option A — `requirePermission({ employee: ['read'] })` + per-section grain + `module.hr-dashboard` (RECOMMENDED, = D-04)
Gate every `hrDashboard.*` procedure on the `employee:read` permission (the same gate the workforce routers already use) — which resolves to **owner + hr_admin + hr_manager + payroll_officer + leave_approver** and no other role, because the base `employee` resource is granted nowhere else `[VERIFIED: roles.ts]`. Layer `module.hr-dashboard` (new, default-off, mirrors `module.workforce-employees`) as the surface kill switch, and require `module.workforce-employees` as the data prerequisite (defense in depth — the employee tables are dark without it). For HR-DASH-03, the per-document expiry rows are filtered by the caller's `employeeFile{A|B|C|D}:read` grants so section B/C items are invisible to roles without that section (C7).

- **Pros:** exactly D-04; no new permission to invent — the `employee` resource *is* the HR surface; excludes finance/ops/admin by construction (they lack the grant); the per-section grain is already role-encoded, so the filter is a lookup, not new policy; a single auditable gate.
- **Cons:** `leave_approver` (a narrow role) also passes `employee:read` — but that is correct (a leave approver legitimately needs headcount/leave context), and its narrow section grants (only A) already restrict what it sees in the doc-expiry widget; `owner` passes (correct — an org owner sees their own HR dashboard). CONTEXT names only hr_admin/hr_manager, but gating any tighter would require a bespoke role check that duplicates the existing permission model.
- **Verdict:** RECOMMENDED. It is D-04, it reuses the shipped RBAC surface, and it makes the per-section grain fall out for free.

### Option B — a bespoke allow-list of role names (`['hr_admin','hr_manager']`) in the procedure
Hard-code the two role names CONTEXT names.

- **Cons:** duplicates the permission model as a string list; drifts when roles are renamed/added; ignores `payroll_officer`/`leave_approver` who legitimately consume HR data; bypasses the section-grain machinery. **Rejected — fragile and off-model.**

### Option C — reuse the staff dashboard `report:read` gate
Mount `hrDashboard` like `dashboard.*`.

- **Cons:** `report:read` is held by `finance_admin`, `payroll_officer`, `ops_manager`, `admin` (via `allPermissions`), and `readonly`-adjacent roles — it would **leak** section-sensitive HR data (discipline, pay, PII-adjacent) to finance/ops. **Rejected — it is the wrong RBAC surface (C5).**

**Recommendation: Option A.** Every plan assumes it.

### The org-scoping + aggregation-correctness fence (the second load-bearing invariant)
Every `hrDashboard.*` read derives `organizationId` from the session (`tenantProcedure` → `ctx.organizationId`) and **never** from client input; every `groupBy`/rollup/date-window `where` spells out `organizationId`. The RED net (97-02) proves: a non-HR role is blocked (`FORBIDDEN`); a cross-org caller's aggregates return nothing (no org-A row leaks into org-B totals); the doc-expiry rows are section-filtered (payroll_officer sees only section C); the namespace is absent / `METHOD_NOT_FOUND` when `module.hr-dashboard` is OFF; and the aggregates are *correct* (headcount buckets sum to the total active count; utilization taken/entitled matches the `LeaveBalance` cache; the probation window buckets partition 14/7/0 at the TZ boundary). Aggregation correctness is a security property here — a wrong `where` is both a bug and a cross-tenant leak.

## Reuse Patterns (VERIFIED)

- **Pattern 1 — rate-limited, tenant-scoped, HR-gated aggregate read.** `const hrDashboardProcedure = tenantProcedure.use(requirePermission({ employee: ['read'] })).use(assertHrDashboardEnabled).use(reportRateLimitMiddleware); hrDashboardProcedure.query(({ ctx }) => ctx.db.employeeProfile.groupBy({ by: ['department'], where: { organizationId: ctx.organizationId }, _count: { _all: true } }))` — mirrors `rateLimitedDashboardProcedure` `[VERIFIED: dashboard.ts:42-44]` but swaps `report:read` → `employee:read` (C5). No client id trusted.
- **Pattern 2 — utilization from the balance cache.** For each active worker-year `LeaveBalance`, `available = entitledMinutes + carryoverMinutes`, `taken = usedMinutes`, `unusedDays = (available - taken) / minutesPerDay`; the under-utilization flag = `unusedDays > 10 && isWithinYearEndWindow(now)`. Reads the materialized cache directly `[VERIFIED: leave.prisma:105-123]` — no ledger re-sum (D-01).
- **Pattern 3 — employee-doc expiry adapter (pure).** `deriveEmployeeDocExpiry(docs, now)` maps each `{ expiresAt, docCategory, section, countryCode }` → `daysUntilExpiryInTz(expiresAt, tzFor(countryCode), now)`, buckets into 90/60/30/expired bands, and DROPS any row whose `section` the caller cannot `employeeFile{A|B|C|D}:read` (C7). Reuses `compliance-policy` `[VERIFIED: expiry.ts:42-50]`; no reminder-cron reuse (C4).
- **Pattern 4 — per-country nationalisation rollup.** Extend `computeSaudizationDashboard` to accept `{ country: 'KSA' | 'UAE', headcount: ManualHeadcountInput, config }` and return the same result shape per country; the rate stays manual-input-only and the band read-through `[VERIFIED: saudization-dashboard.ts:105-114]` (C6). No `EmployeeProfile`-derived rate.
- **Pattern 5 — probation date-window watchlist.** `employeeProfile.findMany({ where: { organizationId, employmentStatus: { in: [ACTIVE, ON_LEAVE] }, probationEndsAt: { gte: startOfTodayTz, lte: addDays(startOfTodayTz, 14) } }, select: { workerId, probationEndsAt, worker: { displayName } } })`, then bucket by `daysUntilExpiryInTz(probationEndsAt, tz, now)` into `{ 0, ≤7, ≤14 }` (D-03).
- **Pattern 6 — dark-mount gating.** `hrDashboard` mounts inside `workforceRouters` (dark under `module.workforce-employees` via `conditionalWorkforceRouters` `[VERIFIED: root.ts:196-198]`) AND each procedure re-asserts `module.hr-dashboard` per request (mirror `assertWorkforceEnabled` `[VERIFIED: employee-time.ts]`). Absent flag → `METHOD_NOT_FOUND`.
- **Pattern 7 — web-vite dashboard widget.** A hook under `components/hr-dashboard/hooks/use-*.ts` is the sole `useTRPC` boundary; a wired section decides loading/empty/error/forbidden; a `*View` is props-only; the page is `Suspense` + `HrDashboardPageContent` `[VERIFIED: reports.tsx:22-106; ARCHITECTURE.md]`. Each widget's empty state maps to its source-surface flag (§ Widget Gating Map).

## Open Questions (resolved with a recommendation)

- **Q1 — the flag.** Reuse `module.workforce-employees` or mint `module.hr-dashboard`? **Resolved:** mint `module.hr-dashboard` (category `module`, `default:false`, owner `workforce-platform`, signoff PENDING) as the **surface** gate, and require `module.workforce-employees` as the **data prerequisite** (the employee tables dark-mount under it). Rationale: an org may run internal workforce management without opening the aggregated HR command surface; a dedicated flag lets it ship dark and flip independently. Both checked (defense in depth). Plan 97-01. (Mirrors P96's `module.employee-portal` decision.)
- **Q2 — namespace.** `hrDashboard.*` vs extend `dashboard.*`/`report.*`? **Resolved:** a dedicated `hrDashboard.*` mounted inside `workforceRouters`. `dashboard.*`/`report.*` gate on `report:read` (wrong RBAC surface — C5) and are not dark-mounted under the workforce flag. Plan 97-03.
- **Q3 — proactive reminder cron (D-03).** Wire probation/under-utilization notifications? **Resolved:** **NO** in v7.0 — ship read-only date-window widgets (they satisfy "auto-surface"). A notification cron adds a digest-throttle dependency and a delivery surface for marginal value this milestone; if built later it MUST reuse the `compliance-reminder-scan` two-pass digest throttle (the v1.0 spam lesson), never per-item. Recorded in Deferred Ideas.
- **Q4 — data capture for the new columns.** `department`/`employmentType`/`contractEndDate`/`probationEndsAt`/`expiresAt`/`docCategory` are added in 97-01 but the dashboard only READS them. Who writes them? **Resolved:** the values are captured by the existing P90 employee-registry edit surface and P91 akta document surface — 97-01 adds the columns to the registry/akta write schemas (a minimal validator + form-field addition) so HR staff can populate them; the dashboard aggregates whatever is populated (nulls fall into an "unspecified" bucket / are excluded from the date-window). Wiring the full registry form UI for these fields is a thin add in 97-01/97-06, not a new phase. The dashboard degrades gracefully when a column is null (Widget Gating Map).
- **Q5 — contract-end vs termination overlap.** `terminatedAt` (admin instant) vs `contractEndDate` (scheduled). **Resolved:** they are distinct (C3) — HR-DASH-01 "by contract-end" buckets `contractEndDate`; a `terminatedAt`-set worker is excluded from active headcount regardless. No conflation.
