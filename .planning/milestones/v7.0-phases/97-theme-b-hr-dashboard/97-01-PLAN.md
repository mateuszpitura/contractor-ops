<objective>
Lay the additive data-layer + flag foundation the HR dashboard reads. RESEARCH corrections C1–C4 established that the widgets' source columns do not exist at HEAD: P90 promoted only `saudizationCategory`/`etat`/`employmentStatus` (+`countryCode`/`terminatedAt`) — NOT `department`, `employmentType` (contract type), or a scheduled `contractEndDate`; there is no `probationEndsAt` anywhere; and `PersonnelFileDocument` has no `expiresAt`/`docCategory`. This plan adds them, mirroring P90's promoted-column + `__`-prefixed-migration pattern exactly.

Four additive, reversible pieces:

1. Promote four nullable, indexed aggregation columns onto `EmployeeProfile` (`department`, `employmentType` via a new `EmploymentType` enum, `contractEndDate @db.Date`, `probationEndsAt @db.Date`) — the columns HR-DASH-01 groups by and HR-DASH-04 windows over.
2. Add `expiresAt @db.Date` + `docCategory` (new `EmployeeDocCategory` enum) to `PersonnelFileDocument` so employee akta docs feed the pure F1 expiry math (HR-DASH-03, C4) — no fork of the reminder engine.
3. Register the `module.hr-dashboard` gate (RESEARCH Open Q1), mirroring `module.workforce-employees`.
4. Extend the employee-registry + akta write-input schemas to accept the new fields (optional) so HR can populate them (RESEARCH Q4).

Purpose: every later plan reads these — the aggregation columns (97-03), the expiry/probation columns (97-04), and the flag gate (all `hrDashboard.*` procedures + the UI nav). This ships GREEN (a foundation gate, not a RED plan). No `hrDashboard` router, no aggregation, no UI in this plan.
</objective>

<threat_model>
## Trust Boundaries
| Boundary | Description |
|----------|-------------|
| Write-input → EmployeeProfile columns | The registry `.strict()` schema is the only write path; the dashboard is read-only, so no new mutation surface is introduced here. |
| Nullable columns → aggregation | All new columns are nullable; the dashboard must treat null as an "unspecified" bucket / exclude from date windows (enforced in 97-03/04), never crash. |

## STRIDE Threat Register
| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-97-01-01 | Tampering | a registry write injecting an out-of-range enum / oversized department string | mitigate | The `.strict()` Zod schema (Task 4) bounds the string + validates the enum; no free-form JSON path. |
| T-97-01-02 | Information Disclosure | the new columns leaking cross-org | mitigate | `EmployeeProfile`/`PersonnelFileDocument` are tenant-owning (absent from `globalModels`) so `withTenantScope` guards every read; the aggregation `where` also spells out `organizationId` (97-02 RED-tests it). |
| T-97-01-03 | Denial of Service | an unindexed groupBy/window scan on a large org | mitigate | Every new column is indexed with an `organizationId` composite (Tasks 1–2); `report-rate-limit` caps the aggregate reads (97-03). |
</threat_model>
