# Phase 87 — UI Design Contract

> Visual and interaction contract for the 1042-S + US worker-classification + Determination-Letter
> + 1099-K-tracker surfaces.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope is deliberately SMALL and EXTENDS shipped surfaces — no net-new shells.** Staff get
> (1) a US worker-classification assessment surface extending the v5.0 classification UI, (2) a
> Determination-Letter generate/download action mirroring the v5.0 SDS generator, (3) a 1042-S
> filing surface extending the P86 tax-filing surface (status pills, consent gate, tokens), and
> (4) an informational 1099-K band on the contractor profile. The portal gains a 1042-S recipient
> PDF download gated on the **same** IRS e-delivery consent already built in P86. Every value below
> is pre-populated from `87-CONTEXT.md` (D-01..D-15), `87-RESEARCH.md`, the **approved P86 UI-SPEC**
> (`86-UI-SPEC.md` — the parent contract this extends), the existing `components.json` (`base-nova`),
> `packages/ui/src/styles/tokens.css`, and the verified v5.0/P85/P86 idioms
> (`classification-tile.tsx`, `generate-sds-button.tsx`, `advisory-banner.tsx`,
> `classification-disclaimer-dialog.tsx`, `tax-form-status-card.tsx`, `step-attest.tsx`,
> `ssn-masked-reveal.tsx`). No new design questions were required.

## Spacing Scale

Declared values (multiples of 4; snap to the repo's semantic vertical-rhythm tokens from `tokens.css`):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, pill icon→label gap (`gap-1`/`gap-1.5`), verdict badge `me-1` icon gap |
| sm | 8px | `--space-stack-tight` (label+control, heading+body, factor question+answer) — `space-y-stack-tight` |
| md | 16px | `--space-card-gap` (rows inside a card/panel, assessment factor rows, classification gate `p-4`) — `space-y-card-gap`; default element spacing |
| lg | 24px | `--space-section-gap` (between top-level page sections) — `space-y-section-gap`; section padding |
| xl | 32px | Layout gaps between major panels (assessment column vs document column) |
| 2xl | 48px | Major section breaks / empty-state vertical padding (`py-12`) |
| 3xl | 64px | Page-level spacing |

**Rhythm rule (inherited):** reach for `--space-stack-tight` / `--space-card-gap` /
`--space-section-gap` (the three named gaps) at page/section level. Raw `space-y-N` / `gap-N`
remains valid only for intrinsic component internals (assessment factor lists, table rows, dialog
bodies, skeletons, stepper internals, the SDS-mirror gate's `space-y-4`) — exactly as the v5.0
classification UI and the P85/P86 surfaces do.

Exceptions: WCAG interactive minimum — all clickable controls (generate-letter / download / transmit
/ override / escalate / consent buttons, consent + override-reason checkboxes and their hit targets)
must meet a **44×44px** minimum touch target. Use `size="sm"` shadcn buttons with adequate padding
or icon buttons sized `size-9`+.

Icon-gap optical exception: `gap-1.5` (6px) is permitted as a sub-4px icon-to-label optical refinement on pill/badge internals only — it is not a structural layout-spacing value (inherited verbatim from the approved P86 contract).

## Typography

Inherited from the established system — exactly the roles P85/P86 use. Two weights only (regular 400 + semibold 600).

| Role | Size | Weight | Line Height | Utility (verified in-tree) |
|------|------|--------|-------------|----------------------------|
| Body | 14px | 400 | 1.5 | `text-sm` (default body for cards/tables/assessment factors/forms) |
| Label | 12px | 400 (600 only for emphasis) | 1.4 | `text-xs text-muted-foreground` (dt labels, rule-set version, completed-at, adviser-verify notes, citations, disclaimer body) |
| Heading | 18px | 600 | 1.2 (`leading-tight`) | `font-display text-lg font-semibold leading-tight` (card/section headings — matches `TaxFormStatusCard` h3) |

**Monospace exception:** TIN/FTIN last-4, EIN, 1042-S income/status/exemption/LOB codes, treaty
article (e.g. `Article 7`), withholding rate (box 3b), amounts (USD), IRS Receipt IDs, rule-set
version string, and 1099-K cumulative-payout / transaction-count figures render in `font-mono`
(matches P85 `dd.font-mono` for `formType` and the P86 `font-mono` for amounts/IDs). This is a
data-legibility affordance, not a 5th type role.

**US classification verdict + AB5 / §530 flags (the load-bearing reconciliation — read carefully):**

The US classification **verdict** mirrors the verified v5.0 `classification-tile.tsx` `toneForOutcome`
mapping (employee=destructive, indeterminate=warning, contractor/independent=success) — a worker
classified *as an employee* is a genuine red-risk verdict, consistent with the IR35 `inside` tone.
**BUT** the **CA AB5 watchlist flag** and the **§530 relief-eligibility flag** are decision-support
*signals*, NOT verdicts — D-04/D-05 are explicit they must read **amber advisory, never alarming red**:

| Element | Tone | Badge / treatment | Token | Rule |
|---------|------|-------------------|-------|------|
| Verdict: likely-contractor (independent) | `success` | `outline` + `border-success/30 bg-success/10 text-success` + `CircleCheck` | `--success` | mirrors `classification-tile` outside/green |
| Verdict: indeterminate | `warning` | `outline` + `border-warning/30 bg-warning/10 text-warning` + `ShieldQuestion` | `--warning` | mirrors `classification-tile` indeterminate |
| Verdict: likely-employee | `destructive` | `outline` + `border-destructive/30 bg-destructive/10 text-destructive` + `ShieldX` | `--destructive` | mirrors `classification-tile` inside/red — the verdict, not the AB5 flag |
| **CA AB5 watchlist flag (auto)** | `warning` | `warning` badge + `AlertTriangle` + tooltip | `--warning` amber | **NEVER `destructive` — advisory watchlist, D-04** |
| **§530 safe-harbor relief flag** | `info` | `info` badge + `ShieldCheck` + tooltip | `--info` blue | relief eligibility, neutral-positive — never red |
| Admin override applied (audit-logged) | `secondary` | `secondary` badge + `PencilLine` | `--secondary` | override is recorded, not a status |

**1099-K informational band (contractor profile) — informational ONLY, never a filing CTA:**

| Band | Tone | Treatment | Token | Rule |
|------|------|-----------|-------|------|
| SAFE | `secondary` / neutral | muted band, no alarm | `--muted` | well below $20,000 + 200 |
| APPROACHING | `warning` | amber band + `TrendingUp` | `--warning` amber | nearing threshold — proactive heads-up, advisory |
| OVER | `warning` | amber band + `AlertTriangle` | `--warning` amber | **NOT `destructive`** — the platform never files 1099-K; this is a courtesy heads-up, never a problem state we own |

**The 1099-K band uses amber at most — never destructive red.** D-10 is explicit: it is purely
informational, never a filing CTA and never a blocking/error state. Crossing $20,000 + 200 is a
heads-up about the settlor's obligation, not our failure.

**Focal point per surface (what the eye lands on first):**
- **US classification assessment (staff):** the sticky advisory banner (amber `role="note"`), then the verdict badge + the scored-factor summary; the AB5 flag and §530 flag sit beside the verdict as amber/blue chips, deliberately not screaming red.
- **Determination letter (staff):** the SDS-mirror approval gate (amber card) → the "Generate determination letter" primary CTA → the generated-document history row.
- **1042-S filing (staff):** the current IRIS submission-status pill, then the "Generate 1042-S batch" / "Download validated 1042-S XML" primary CTA, with the per-recipient treaty-rate-vs-30% column.
- **1099-K band (profile):** a compact informational band/badge on the contractor profile showing cumulative payout + transaction count vs the {taxYear} threshold — calm, never alarming.
- **Recipient PDF (portal):** the affirmative consent checkbox + the "Consent and download my 1042-S" affirm block (reuses the P86 consent step).

## Interaction & State Contract (mandatory — checker validates)

Every **wired section** must branch all four states (verified pattern in `tax-form-status-card.tsx`
and `classification-tile.tsx`):

| State | Treatment |
|-------|-----------|
| Loading | `Skeleton` placeholders shaped like the content; container carries `aria-busy` + `aria-live="polite"` (the v5.0 `ClassificationTileSkeleton` sets `aria-busy="true"`) |
| Empty | centered `py-12`/`py-6` block: `font-display` heading + `text-muted-foreground` body + (where relevant) the primary CTA. Copy from the contract above. |
| Error | `role="alert"` `text-destructive` message stating problem + next step + an outline "Reload"/retry button (data preserved on retry, never silently dropped) |
| Loaded | the presentational content |

**Status-change announcements (WCAG):** when a 1042-S IRIS filing status, a batch-generation
outcome, a **classification verdict**, or an **AB5-flag change** changes, announce it via an
`aria-live="polite"` region (use `assertive` only for a Rejected/error transition). Classification
result and AB5-flag changes are explicitly named here because they are decision-relevant — mirrors
the `aria-busy`/`aria-live` already used by P85/P86 and v5.0.

**Keyboard + focus:** all actions reachable and operable by keyboard; visible `--ring` teal focus on
every focusable control; the file-upload field, consent checkbox, override-reason textarea, and the
letter-approval checkbox are real native inputs with associated `<Label htmlFor>`;
reveal/override/escalate/generate buttons' accessible name == visible text (WCAG Label-in-Name — the
P84/P85/P86 invariant). The blocking disclaimer `AlertDialog` has its dismiss path disabled — it
cannot be bypassed by keyboard or pointer (verified in `classification-disclaimer-dialog.tsx`).

**Advisory-not-verdict invariant (D-05 — the load-bearing one for this phase):** the US
classification surface reads as **decision-support, never a legal verdict**. (a) The sticky amber
`ClassificationAdvisoryBanner` (`role="note"`) is always present on the assessment/result surface.
(b) The blocking `ClassificationDisclaimerDialog` must be acknowledged before the outcome is shown.
(c) The **AB5 watchlist flag is amber `warning`, never `destructive` red** — it is a "review with your
adviser" flag, not a determination. (d) The **§530 flag is `info` blue** — relief eligibility, neutral.
(e) Every result and the determination letter carry the adviser-verify note. The verdict badge itself
may use the red `inside`-equivalent tone (a likely-employee outcome is a genuine risk signal,
consistent with IR35), but it is framed by the banner + disclaimer + adviser-verify note as advisory.

**1099-K informational-only invariant (D-10):** the 1099-K band offers **no filing affordance** — no
"file", "generate", or "fix" button. It is a read-only band. The OVER state uses amber at most and
says "your payment settlor — not this platform — files any 1099-K". Never `destructive`, never an
error/`role="alert"` treatment.

**Never-hard-block invariant (1042-S withholding, D-07):** a recipient without a complete W-8 chain
is reported at the 30% statutory rate (amber caption), **never blocked** from filing. The 30%-vs-treaty
distinction is surfaced as an advisory caption, not a red gate. Routing a foreign W-8 recipient to
1042-S vs a US W-9 recipient to 1099-NEC derives from the form on file (P85) — a missing/invalid
W-form + US-source income surfaces an escalation, never a silent skip.

**Human-in-the-loop (D-15):** the 1042-S batch surface is explicitly *review-before-file*. "Generate
1042-S batch" produces a reviewable withholding summary; filing (download XML / transmit) is a
separate, deliberate action. The classification override is reason-required + audit-logged. No UI
affordance auto-files or silently overrides.

## IRS Electronic-Delivery Consent Contract (portal — Pub 1179 §4.6, D-09 — SAME GATE AS P86)

The 1042-S recipient PDF reuses the **exact** P86 affirmative IRS e-delivery consent step — this is
not a new interaction, it is the same `step-edelivery-consent.tsx` + `use-edelivery-consent.ts`
gating a second form type. Model faithfully, mirroring the P85 ESIGN attestation idiom
(`step-attest.tsx`):

1. **Affirmative opt-in only.** A real native `<input type="checkbox">` (unchecked by default),
   labeled with explicit consent language: the recipient consents to receive their 1042-S
   electronically, confirms they can access the PDF format, and understands they may request a paper
   copy and may withdraw consent. The "Consent and download" button is **disabled until the checkbox
   is checked** — no implied/pre-checked consent.
2. **Server-derived audit (never client-supplied):** consent timestamp, IP, and actorId are 100%
   derived server-side (`deriveClientIp(ctx.headers)` / `ctx.contractorId` / `new Date()`), exactly
   as P85 ESIGN attestation and the P86 consent step do. The client schema omits all three.
3. **Consent gates download.** With consent → the 1042-S recipient PDF download is available. Without
   consent → show the paper-copy messaging; the download is not offered. **No portal seat** (common
   for foreign W-8 recipients) → flag for paper/manual delivery (no-portal-seat messaging above). The
   immutable PDF is archived to R2 either way (D-09) — the gate is on *furnishing*, not generation.
4. **Withdrawable.** A "Withdraw electronic-delivery consent" action (destructive-confirm copy above)
   flips back to the paper-copy state; re-consent is allowed at any time.
5. **No full TIN/FTIN.** The PDF and any on-screen recipient identifier show TIN/FTIN **last-4 only**
   (Pub 1179 masking + the P84/P85/P86 `SsnMaskedReveal` invariant). Full value never reaches the DOM,
   the snapshot, or any log.
