## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client upload → ingestAchReturnFile | An operator uploads an externally-produced NACHA return file (untrusted text) across the tRPC boundary. |
| procedure → run items | The flip-to-FAILED transition is gated + tenant-scoped; the actual mutation + audit live in 88-10's applyAchReturns. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-88-11-01 | Tampering | uploaded return file | mitigate | .strict() Zod input with a bounded returnFileText length; parseNachaReturnFile parses defensively (88-10) and never trusts the payload shape. |
| T-88-11-02 | Elevation of privilege | who can flip payouts | mitigate | requirePermission({payment:['export']}) + assertUsExpansionEnabled + tenant scope — mirrors initiatePayout's gate. |
| T-88-11-03 | Denial of service | oversized upload | mitigate | returnFileText max length cap in the Zod schema bounds the parse cost. |
| T-88-11-04 | Repudiation | untracked ingestion | mitigate | applyAchReturns writes one masked audit row per transition; the ingestion delegates to it (no silent flip). |
| T-88-11-05 | Spoofing | forged live webhook (future) | mitigate | The live return-webhook is a documented deferred seam (PayoutInitiationAdapter.handleWebhook must safeParse + verify signature) — NOT enabled here; programmatic ACH stays dark. |
| T-88-11-06 | Information disclosure / operator safety | mis-uploaded file applied to the wrong run | mitigate | The returned `unmatched` count (88-10) surfaces a wrong-run/mis-uploaded file to the operator instead of an indistinguishable all-zeros no-op. |
| T-88-11-SC | Tampering | npm/pip/cargo installs | accept | No package installs — the ingestion reuses the hand-rolled 88-10 service. |
</threat_model>
