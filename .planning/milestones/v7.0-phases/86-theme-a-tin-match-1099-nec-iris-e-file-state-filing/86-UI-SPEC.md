# Phase 86 — UI Design Contract

> Visual and interaction contract for the year-end US information-return loop UI surface.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope is deliberately SMALL and EXTENDS existing P85 surfaces — no net-new shells.** Staff
> get a 1099-NEC batch + IRIS-filing + TIN-mismatch surface; the portal gains one IRS
> electronic-delivery consent step gating the Copy-B PDF download. Every value below is
> pre-populated from `86-CONTEXT.md` (D-01..D-20), `86-RESEARCH.md`, the existing
> `components.json` (`base-nova`), `packages/ui/src/styles/tokens.css`, and the verified P85
> idioms (`tax-form-status-card.tsx`, `tax-form-wizard.tsx`, `step-attest.tsx`,
> `ssn-masked-reveal`). No new design questions were required.

## Spacing Scale

Declared values (multiples of 4; snap to the repo's semantic vertical-rhythm tokens from `tokens.css`):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, pill icon→label gap (`gap-1`/`gap-1.5`) |
| sm | 8px | `--space-stack-tight` (label+control, heading+body pairs) — `space-y-stack-tight` |
| md | 16px | `--space-card-gap` (rows inside a card/panel) — `space-y-card-gap`; default element spacing |
| lg | 24px | `--space-section-gap` (between top-level page sections) — `space-y-section-gap`; section padding |
| xl | 32px | Layout gaps between major panels |
| 2xl | 48px | Major section breaks / empty-state vertical padding (`py-12`) |
| 3xl | 64px | Page-level spacing |

**Rhythm rule (inherited):** reach for `--space-stack-tight` / `--space-card-gap` /
`--space-section-gap` (the three named gaps) at page/section level. Raw `space-y-N` remains
valid only for intrinsic component internals (table rows, dialog bodies, skeletons, stepper
internals) — exactly as the P85 status-card and wizard do.

Exceptions: WCAG interactive minimum — all clickable controls (download / upload / escalate /
resolve buttons, consent checkbox + its hit target) must meet a **44×44px** minimum touch
target. Use `size="sm"` shadcn buttons with adequate padding or icon buttons sized `size-9`+.

## Typography

Inherited from the established system — exactly the roles P85 uses. Two weights only (regular 400 + semibold 600).

| Role | Size | Weight | Line Height | Utility (verified in-tree) |
|------|------|--------|-------------|----------------------------|
| Body | 14px | 400 | 1.5 | `text-sm` (default body for cards/tables/forms) |
| Label | 12px | 400 (600 only for emphasis) | 1.4 | `text-xs text-muted-foreground` (dt labels, captions, adviser notes) |

schema VersionNum render in `font-mono` (matches P85 `dd.font-mono` for `formType`). This is a
data-legibility affordance, not a 5th type role.

## Color

60/30/10 split inherited from `tokens.css` (OKLCh). Accent (teal `--primary`) is reserved, never sprayed on every interactive element.

| Role | Value (token) | Usage |
|------|---------------|-------|
| Dominant (60%) | `--background` `oklch(0.985 0.003 85)` | Page background, route shell surfaces |
| Secondary (30%) | `--card` `oklch(1 0 0)` / `--secondary` / `--muted` | Cards (batch panel, filing-status card, mismatch list, consent card), section chrome, table surfaces |
| Accent (10%) | `--primary` Deep Teal `oklch(0.44 0.145 178)` | Primary CTAs only (see reserved list); Stepper active indicator; focus ring (`--ring`) |
| Destructive | `--destructive` `oklch(0.52 0.23 18)` | Rejected ack status, consent-withdrawal confirm, validation-error text (`role="alert"`) |

**Accent (teal `--primary`) reserved for — explicit list, nothing else:**
- The single primary CTA per surface: "Generate 1099-NEC batch" (batch surface), "Download validated XML" / "Upload IRS acknowledgement" (filing surface), the consent affirm button (portal).
- The reui Stepper active-step indicator (portal consent step, if rendered inside the existing wizard stepper).
- Focus-visible ring on all focusable controls (`--ring` = teal).

Everything else uses neutral/secondary/outline variants. Secondary actions ("Download per-state
CSV", "Escalate mismatch", "Mark resolved", "Back", "Reload") use `variant="outline"` or `variant="secondary"`.

**Semantic status colors (NOT part of the 10% accent — these are state signaling, reused from `tokens.css` + P85 badge variants):**

| Filing / mismatch state | Badge variant (verified in `badge.tsx`) | Icon (lucide) | Token |
|-------------------------|------------------------------------------|---------------|-------|
| Accepted | `success` | `CheckCircle2` | `--success` green |
| Accepted with Errors | `warning` | `AlertTriangle` | `--warning` amber |
| Partially Accepted | `warning` | `AlertTriangle` | `--warning` amber |
| Transmitted / Processing | `info` | `Clock` / `Loader2` | `--info` blue |
| Built | `info` | `FileText` | `--info` blue |
| Downloaded-for-manual-upload | `secondary` | `Download` | `--secondary` |
| Rejected | `destructive` | `XCircle` | `--destructive` |
| Not Found | `secondary` | `Info` | `--secondary` |
| TIN mismatch (recipient) | `warning` | `AlertTriangle` | `--warning` amber |
| TIN matched | `success` | `CheckCircle2` | `--success` |
| Backup-withholding flag set | `warning` | `AlertTriangle` | `--warning` amber (advisory, never a destructive/blocking red) |
| CORRECTED (supersede) | `secondary` | `XCircle` | prior row marked superseded |

Mismatch and backup-withholding use **amber `warning`, never `destructive` red** — D-12 is
explicit: a TIN mismatch is *never a hard block*; the 1099 still generates. Red is reserved for
genuine Rejected ack + destructive confirmations only.

**Focal point per surface (what the eye lands on first):**
- **Batch (staff):** the "Generate 1099-NEC batch" primary CTA paired with the batch-summary card (recipients above/below the {taxYear} threshold + totals).
- **Filing (staff):** the current IRIS submission-status pill, then the "Download validated XML" primary CTA.
- **Mismatch (staff):** the mismatch-count summary heading, then the recipient list — amber, deliberately not alarming red.
- **Consent (portal):** the affirmative consent checkbox + the "Consent and download my 1099-NEC" affirm block.

## Interaction & State Contract (mandatory — checker validates)

Every **wired section** must branch all four states (verified pattern in `tax-form-status-card.tsx`):

| State | Treatment |
|-------|-----------|
| Loading | `Skeleton` placeholders shaped like the content; container carries `aria-busy` + `aria-live="polite"` |
| Empty | centered `py-12`/`py-6` block: `font-display` heading + `text-muted-foreground` body + (where relevant) the primary CTA. Copy from the contract above. |
| Error | `role="alert"` `text-destructive` message stating problem + next step + an outline "Reload" / retry button (data preserved on retry, never silently dropped) |
| Loaded | the presentational content |

**Status-change announcements (WCAG):** when an IRIS filing status, TIN-match result, or
batch-generation outcome changes, announce it via an `aria-live="polite"` region (use
`assertive` only for a Rejected/error transition). Mirrors the `aria-busy`/`aria-live` already
used by P85.

**Keyboard + focus:** all actions reachable and operable by keyboard; visible `--ring` teal
focus on every focusable control; the file-upload field and consent checkbox are real native
inputs with associated `<Label htmlFor>`; reveal/escalate/resolve buttons' accessible name ==
visible text (WCAG Label-in-Name — the P84/P85 invariant).

**Never-hard-block invariant (D-12):** the TIN-mismatch surface presents escalate/resolve and
shows the auto-set backup-withholding flag, but offers **no path that blocks 1099 generation**.
The mismatch is surfaced as amber advisory, never a red gate.

**Human-in-the-loop (D-05):** the batch surface is explicitly a *review-before-file* posture.
"Generate batch" produces a reviewable summary; filing is a separate, deliberate action. No UI
affordance auto-files. The notify-only year-end cron has no UI beyond an in-app/notification
reminder (no auto-generation control).

## IRS Electronic-Delivery Consent Contract (portal — Pub 1179 §4.6, D-09)

This is a real US compliance interaction — model it faithfully, mirroring the P85 ESIGN
attestation idiom (`step-attest.tsx`):

1. **Affirmative opt-in only.** A real native `<input type="checkbox">` (unchecked by default),
   labeled with explicit consent language: the recipient consents to receive their 1099-NEC
   electronically, confirms they can access the PDF format, and understands they may request a
   paper copy and may withdraw consent. The "Consent and download" button is **disabled until
   the checkbox is checked** — no implied/pre-checked consent.
2. **Server-derived audit (never client-supplied):** consent timestamp, IP, and actorId are
   100% derived server-side (`deriveClientIp(ctx.headers)` / `ctx.contractorId` / `new Date()`),
   exactly as P85 ESIGN attestation does. The client schema omits all three.
3. **Consent gates download.** With consent → the Copy-B PDF download is available. Without
   consent → show the paper-copy messaging; the download is not offered. The immutable PDF is
   archived to R2 either way (D-09) — the gate is on *furnishing*, not on generation.
4. **Withdrawable.** A "Withdraw electronic-delivery consent" action (destructive-confirm copy
   above) flips back to the paper-copy state; re-consent is allowed at any time.
5. **No full TIN.** The PDF and any on-screen recipient identifier show TIN **last-4 only**
   (Pub 1179 masking + the P84/P85 SsnMaskedReveal invariant). Full TIN never reaches the DOM.
