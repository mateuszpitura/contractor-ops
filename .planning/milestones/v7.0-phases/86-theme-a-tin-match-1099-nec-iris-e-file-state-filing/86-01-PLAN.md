<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| IRS SOR → repo schema-bundle | Externally-downloaded XSDs become a build input; a swapped/tampered schema would silently change validation behavior |
| bundled XSD → libxmljs2 (downstream) | XSDs must not reference external `schemaLocation` URLs (SSRF) — bundle is self-contained |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-86-01-01 | Tampering | bundled IRS XSDs | mitigate | `verify-iris-schema-checksums.ts` pins SHA-256 of every `.xsd`; CI fails on any swap or unlisted file. |
| T-86-01-02 | Information Disclosure | XSD `<xs:import schemaLocation="http://...">` | mitigate | Bundle is self-contained; downstream validator (Plan 04) uses `{ nonet: true }` so no network fetch is possible regardless. Note this invariant in the bundle README. |
| T-86-01-03 | Repudiation | wrong-year schema bundled | mitigate | `source.txt` records the exact TY + posted date; `IrisSubmission` (Plan 02) records the schema VersionNum/VersionDt it was built against (Pitfall 4). |
| T-86-01-SC | Tampering | npm/pip/cargo installs | accept | Zero NEW external deps (RESEARCH § Package Legitimacy Audit: all GA-path packages already in workspace; `fast-xml-parser`/`libxmljs2` are reused verbatim, not freshly installed). No legitimacy checkpoint required. |
</threat_model>
