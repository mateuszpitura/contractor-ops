## Screen 1 — 4-Section Personnel File Shell (AKTA-01)

### Layout

A single route (mounted under the employees domain, e.g.
`pages/dashboard/employees/[workerId]/personnel-file.tsx` — thin page, `Suspense` +
`PersonnelFile` wired section per the web-vite layering contract). Page structure, top to bottom:

1. **Header** (Display 20px/600): employee full name + `Badge` jurisdiction chip (PL/DE/UK/US) +
   employment-status chip (Active / Terminated {date}, the latter feeding the AKTA-02 retention
   anchor display). `space-y` gap: `lg` (24px) below header.
2. **Retention summary strip** (compact, one line per jurisdiction's overall posture — see Screen
   3) — collapsed by default, expandable.
3. **Four section cards**, each a `Card` wrapping a `Collapsible` (default: unlocked sections
   start **expanded**; locked sections render collapsed-and-inert — see States below). Gap
   between cards: `lg` (24px).
4. Each unlocked section card's `CollapsibleContent` composes `DocumentList`/`DocumentCard` from
   `components/documents/*` **unmodified** — do not fork that family.

### Per-section states (mandatory — all four MUST be distinguishable at a glance)

| State | Trigger | Visual treatment |
|-------|---------|-------------------|
| **Loading** | Initial fetch of that section's document list in flight | `Skeleton` rows reusing the exact `DocumentList` loading pattern (`document-list.tsx:20-34` — 3 skeleton rows, `size-12` icon placeholder + 2 text lines). Card header (title/lock icon) renders immediately — only the body skeletons. |
| **Locked** | Caller lacks `employeeFileX:read` for this section (server-enforced — see RBAC contract below) | Card header still renders (section title is NOT hidden — title is non-sensitive metadata). `CollapsibleTrigger` replaced with a non-interactive disabled element: `Lock` icon (lucide) + `Badge` using `--status-blocked` token, text "Section locked". Card body NEVER mounts — no document count, no skeleton, no children at all (avoids leaking "N documents exist" via count). `aria-disabled="true"` on the header row; trigger button NOT `tabIndex={-1}`-removed from focus order entirely — it remains reachable via Tab (so screen-reader users discover the section exists and is locked) but activating it is a no-op (no `onClick` handler, `disabled` attribute set). |
| **Empty** | Caller has read access, zero documents in the section | `DocumentList` empty-state pattern reused verbatim (`document-list.tsx:36-44` — `FileText` icon at half opacity + heading + body, see Copywriting Contract). Upload CTA shown inline below the empty message (not just at page level) so the section is independently actionable. |
| **Error** | That section's document-list query failed (network/5xx) — **must be section-scoped, not a full-page error**, since other sections may have loaded successfully | Card body shows an inline error block: `AlertTriangle` icon, "Couldn't load this section" heading, body text, `Button variant="outline" size="sm"` "Retry" that re-triggers only that section's query. Other section cards remain fully interactive. |
| **Populated** | Caller has read access, ≥1 document | `DocumentList` + `DocumentCard` reused unmodified, each card showing the existing virus-scan/version/preview/delete affordances. |

### RBAC visual contract (D-02 — enforced at the permission layer, not filtered client-side)

- The hook (`use-personnel-file.ts`) calls a procedure that returns, **per section**, one of:
  `{ status: 'locked' }` (no document payload at all) or `{ status: 'unlocked', documents: [...] }`.
  The wired section must branch on `status` — **never** receive all four sections' documents and
  hide some in the view layer (BFLA hole called out in RESEARCH.md Pitfall 3).
- A negative-state visual test is part of the contract: render the shell as `payroll_officer` and
  assert section B (discipline) renders the **Locked** card, never the Empty or Populated card.
- `owner` is never auto-granted any section (BFLA fence, D-02) — if an `owner`-role staff member
  has zero `employeeFileX` grants, **all four** sections render Locked. This is a deliberately
  conspicuous state (4 locked cards in a row) so the absence of the grant is visually undeniable,
  not silently empty.

### Section display labels (seeded, adviser-verify — see i18n Contract for the key shape)

| Jurisdiction | Section A | Section B | Section C | Section D |
|---|---|---|---|---|
| PL | Dane rekrutacyjne (cz. A) | Przebieg zatrudnienia (cz. B) | Ustanie zatrudnienia (cz. C) | Odpowiedzialność porządkowa (cz. D) |
| DE | Stammdaten | Vertrag & Entgelt | Bescheinigungen | Abmahnungen / Sonstiges |
| UK | Personal & recruitment | Employment & pay | Certificates & references | Disciplinary & performance |
| US | I-9 & work authorization *(stored separately per ICE guidance — flag in copy)* | Employment & pay | Certifications & training | Disciplinary & performance |

All four are `[ASSUMED]`/adviser-verify per RESEARCH.md A5/A6/A7 — render every section label with
a small inline info affordance (`Tooltip` + `Info` lucide icon, 12px) carrying the adviser-verify
note from the Copywriting Contract. DE/UK have **no statutory section structure** (A6) — their
labels must visually read as organizational groupings, not cite a law (no citation chip on the
section header for DE/UK; PL/US sections DO carry a citation chip, since those ARE statutory).

## Screen 2 — Document Upload + Classification (AKTA-04)

### Upload flow (staff-side, from inside an unlocked section card)

1. Staff clicks "Upload document" (inside a section card or from the section's empty state) →
   reuses `DropZone`/`FileUpload` (`components/documents/drop-zone.tsx`) unmodified, with
   `entityType="personnelFile"`, `entityId={workerId}`. Uploader picks a `DocumentType` from the
   existing type picker (do not invent a new type list) — this feeds the deterministic taxonomy
   map (D-07a) as the first classification signal.
2. **Immediately** after upload completes (before classification resolves), the document appears
   in its eventual section (or a neutral "Classifying…" placeholder if the section is not yet
   known) — **never block the upload UI waiting on AI** (D-07 / RESEARCH Pitfall 5).
3. Classification outcome renders as a **badge on the document card itself** (not a separate
   screen) — three variants:

| Outcome | Badge | Color token | Confidence shown? |
|---------|-------|-------------|---------------------|
| Deterministic taxonomy hit | "Classified to {sectionLabel}" | `--status-success` | No (no AI involved, no confidence number to show) |
| AI auto-classified (≥ confidence threshold) | "Classified to {sectionLabel}" + `ConfidenceBadge`-style percentage | `--status-success` | Yes — reuse `getConfidenceConfig` thresholds (>90 success / 70–90 warning / <70 destructive — already shipped in `confidence-badge.tsx:19-39`), i18n-wrapped per the Reuse Map note above |
| Routed to admin (ambiguous / below threshold / killswitch off / Unleash unreachable) | "Pending classification review" | `--status-info` | No — uploader never sees an AI confidence number for a doc that didn't clear the bar; that detail belongs only to the admin queue |

4. A document `status: PENDING_REVIEW` is **visible in its section's list as a distinct row
   state** (not hidden until approved) — muted/dashed border treatment + the info badge above, so
   staff can see "this is in the queue," consistent with "visibly gated, not hidden" used for
   locked sections.

### Admin classify review queue (new surface — mirrors `ApprovalQueueTable` + `UploadReviewDialog`)

- **Table** (`personnel-classify-queue/data-table.tsx`): columns — Employee name, Jurisdiction,
  Document (filename + type icon, reuse `getFileIcon` from `document-card.tsx`), Uploaded date,
  AI guess (section + confidence badge, or "—" if AI was skipped), Actions (Approve / Reject row
  buttons, mirroring `RowActions` in `approval-queue/data-table.tsx:118-146`).
- **States:** Loading → `WorkbenchDataTable` built-in skeleton; Empty → "Nothing pending review" /
  "All uploads have been classified."; Error → table-level error row with Retry (same pattern as
  Screen 1's section error, scoped to the whole queue this time since it's a single list).
- **Approve dialog** (`personnel-classify-review-dialog.tsx`, reusing `UploadReviewDialog`'s
  `Tabs` + `DialogBody`/`DialogFooter` shell verbatim):
  - **Approve tab:** a `Select` of the 4 section labels (jurisdiction-resolved per that employee),
    pre-selected to the AI's top guess when one exists; confirm button "Approve & assign section"
    (disabled until a section is chosen).
  - **Reject tab:** closed-enum reason `Select` + optional free-text `Textarea`, mirroring
    `UploadReviewDialog`'s reject tab exactly (`REJECT_REASON_CATEGORIES` pattern) — new reason
    enum values appropriate to mis-routed personnel documents (e.g. "wrong employee", "not a
    personnel document", "duplicate", "illegible/needs re-upload").
  - Confirm button uses `variant="destructive"` only for Reject, default `variant="default"`
    (accent teal) for Approve — matches the existing dialog's variant split.
- **Bulk actions:** out of scope for v1 of this phase (classify-review is lower volume than
  timesheet approvals) — ship row-level actions only; do not add a bulk bar unless the planner
  finds volume evidence otherwise.

## Screen 3 — Retention + Erasure View (AKTA-02 / AKTA-03)

### Retention summary (per-section, surfaced both inline-on-section-card and in a dedicated panel)

- Each section card header carries a small retention chip (right-aligned, 12px Label size):
  - **Active employee, no termination event:** "Retained while employed" (neutral/no token —
    this is normal-course, not a warning).
  - **Terminated, clock running:** "Retained until {date}" using `--status-warning` once inside
    e.g. the final 90 days of the window (visual urgency only near expiry — do not alarm-color a
    10-year-out cutoff), otherwise neutral `--muted-foreground` text.
  - Clicking the chip expands an inline citation panel: statutory citation text (Body 14px) +
    the adviser-verify info icon/tooltip from the Copywriting Contract. PL/US sections show a
    real citation (KP art. 94⁵ / 8 CFR 274a.2); DE/UK sections that are NOT inherently statutory
    (A6) show the citation of the underlying retention rule that DOES apply to that grouping
    (e.g. DE tax/payroll retention) — never fabricate a citation for an organizational-only
    grouping; if no citation applies, omit the chip's citation row entirely rather than inventing
    one.
- A page-level **Retention summary panel** (collapsed by default, below the header) lists all 4
  sections' cutoff dates in one place for an HR admin doing a compliance sweep — reuses the same
  citation-row treatment, no new visual language.

### RODO/GDPR erasure flow (per-employee, staff-initiated)

1. Entry point: "Request erasure" primary-accent button, page-level (not per-section — the
   request always targets the whole `workerId`; the *response* is what's per-section).
2. Clicking opens a confirmation `AlertDialog` (reuse the exact pattern from
   `document-card.tsx:203-222`): title "Request data erasure for {employeeName}?", body explains
   this evaluates every section against its retention window and may not erase everything
   immediately, confirm button `variant="destructive"` (erasure is consequential, even though
   partial) labeled "Request erasure".
3. **Result view** — a per-section disposition list, rendered as a simple stacked list (not a
   table — only 4 rows):

   | Row | Visual |
   |-----|--------|
   | Erased section | `CheckCircle2` icon, `--status-success`, "{sectionLabel}: Erased" |
   | Retained section | `ShieldAlert` icon (NOT `Lock` — distinct from the RBAC-locked icon to avoid confusing "I can't see this" with "this is legally held"), `--status-warning`, "{sectionLabel}: Retained until {date} — {citation}" |

4. **Page-level summary banner above the list** — this is the criterion-#3-critical element:
   - `fullErasureClaimed === true` → success banner ("All sections erased"), `--status-success`,
     `CheckCircle2`.
   - `fullErasureClaimed === false` → warning banner (the partial-erasure copy from the
     Copywriting Contract), `--status-warning`, `ShieldAlert`. **This banner must render even if
     only one of four sections is retained** — never collapse to a success banner with an
     asterisk. The banner is the single most legally load-bearing pixel in this phase; the checker
     should treat any wording here that could be read as "fully erased" while a hold exists as a
     contract violation, not a copy nitpick.
5. Below the disposition list, the same adviser-verify legal note from the Copywriting Contract
   renders in the established amber `role="note"` banner — this is the per-employee erasure
   instance of the Standing Constraint (REQUIREMENTS.md line 26).

## Accessibility & RTL Contract

- **Keyboard:** every interactive element (collapsible triggers, upload CTA, retry buttons, queue
  row actions, dialog controls) reachable via Tab in DOM order; locked-section triggers remain
  reachable (not `tabIndex={-1}`) but are inert (`disabled`, `aria-disabled="true"`) so assistive
  tech announces "Section locked, dimmed/disabled" rather than skipping it silently.
- **Focus:** dialogs (classify-review, erasure-confirm) trap focus per the shared `Dialog`/
  `AlertDialog` primitives (already WCAG-compliant via Radix) — no custom focus management needed.
- **Semantic HTML:** section cards use `<section aria-labelledby={headingId}>` with an `<h3>` or
  `<h4>` heading per section (matches `classification-documents-panel.tsx:44-50` precedent); the
  page header is an `<h1>` or `<h2>` depending on route nesting.
- **Contrast:** all status-token text/background pairs (`--status-*-fg` on `--status-*-bg`) are
  pre-verified WCAG AA in `status.css` — do not introduce ad hoc colors; the one exception (amber
  legal banner) reuses the existing `bg-amber-50 text-amber-800` pair already shipped and presumed
  accessible from prior phases.
- **RTL (ar):** use logical Tailwind utilities exclusively in new code — `ms-`/`me-` (margin),
  `ps-`/`pe-` (padding), `start-`/`end-` (position), never `ml-`/`mr-`/`left-`/`right-`. This
  matches the existing convention (`alert.tsx` `[&>svg]:start-4`, `approval-queue` `me-1.5`). Icons
  used in this phase (`Lock`, `ShieldAlert`, `CheckCircle2`, `AlertTriangle`, `Info`) are
  symmetric — no mirroring needed. Section-card collapsible chevrons (if the shadcn `Collapsible`
  trigger renders a directional affordance) must flip automatically under `[dir="rtl"]` — verify
  against the existing `Collapsible` component's CSS, do not hardcode a left/right chevron.
- **Loading/empty/error states are mandatory on every section, the classify queue, and the
  erasure result view** — no surface in this phase may ship with only the happy path.
