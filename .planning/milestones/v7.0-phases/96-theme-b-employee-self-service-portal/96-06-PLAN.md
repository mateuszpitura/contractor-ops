<objective>
Build the manager backend surface (EMP-PORTAL-03): a `portalManager` namespace whose every read/approval is server-side scoped to the caller's direct reports via the reporting-line edge (96-01), flipping the 96-03 manager-IDOR + manager-approve RED tests GREEN.

Purpose: this is the second security surface. A portal manager is NOT a staff `leave_approver` User (RESEARCH C2/D-03) — so **authorization is the reporting-line edge**, while **execution reuses the shared approval-shared state machine + ledger**. Reports are resolved from `managerWorkerId = ctx.workerId`, never client input; a manager action first asserts the target is a report. The whole namespace sits behind `portalManagerProcedure` (asserts ≥1 report) + `module.employee-portal`.
</objective>

<threat_model>
## Trust Boundaries
| Boundary | Description |
|----------|-------------|
| Manager → reports | Reports are server-derived; a report id is validated against the edge before any read/write. |
| Portal manager vs staff approver | Two authorization paths over one leave request; each acts only in its scope; the shared transition is idempotent on state. |

## STRIDE Threat Register
| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-96-06-01 | Elevation of Privilege | a manager approving for a non-report | mitigate | `assertIsDirectReport` before the transition (Tasks 1,2); `portal-manager-idor`/`approve` GREEN. |
| T-96-06-02 | Information Disclosure | a manager reading a non-report's leave/time/docs | mitigate | Reports server-derived; reads scoped to the reports set (Task 2). |
| T-96-06-03 | Elevation of Privilege | a non-manager reaching the manager surface | mitigate | `portalManagerProcedure` asserts ≥1 report (96-02); a non-manager gets no namespace. |
| T-96-06-04 | Repudiation | an unaudited approval | mitigate | `writeAuditLog` in-tx on approve/reject (Task 2); `lint:audit-log` green. |
| T-96-06-05 | Tampering | double-action across the portal + staff approver paths | mitigate | The shared transition is idempotent on request state; a re-action is a typed no-op/conflict (Open Q5). |
</threat_model>
