<objective>
Add per-tier rate limiting (INTEG-AUTH-04) as a post-auth monthly request quota that composes with the
existing pre-auth burst limiter. A single tier-limits table defines the monthly request quota (Starter
1000 / Pro 10000 / Enterprise unlimited) AND the webhook-subscription caps (1/5/unlimited — defined here,
enforced in Phase 100). A tRPC middleware `enforceApiTierQuota` in `apiKeyTenantProcedure` resolves the
org's tier (Redis-cached `getSubscription`) and increments an Upstash monthly counter, throwing 429 when
the tier quota is exceeded; Enterprise short-circuits as unlimited. The flat 100/min burst limiter stays
untouched (two limiters, two jobs).

Output: the tier-limits table + the monthly-counter service + the quota middleware wired into the chain +
the RED quota test GREEN.
</objective>

## Trust Boundaries
| Boundary | Description |
|----------|-------------|
| API key request → monthly quota | An over-tier consumer must be throttled by billing tier, not just by burst |
| Upstash outage → quota | A counter backend outage must not silently grant unlimited access in prod |

## STRIDE Threat Register
| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-99-03-01 | Denial of Service | Consumer exceeds tier quota | mitigate | Post-auth monthly counter → 429 when over the tier limit |
| T-99-03-02 | Elevation of Privilege | Counter fail-open grants unlimited in prod | mitigate | In-memory fallback is non-prod only + documented; prod requires Upstash (conditional-skip test + .env note) |
| T-99-03-03 | Tampering | Wrong reset window double-counts / never resets | mitigate | Calendar-month fixed window (YYYY-MM key + month-end TTL); Enterprise writes no counter |
| T-99-03-SC | Tampering | npm installs | accept | No new deps (reuse @upstash/redis already present) |
</threat_model>
