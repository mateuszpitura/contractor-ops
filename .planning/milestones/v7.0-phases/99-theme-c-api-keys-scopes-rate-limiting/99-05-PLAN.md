<objective>
Add key rotation with a grace period (INTEG-AUTH-01 "rotates" + INTEG-AUTH-03 rotation flow backend). A
`rotate` mutation issues a fresh `co_live_*` key inheriting the old key's name/scopes/actingUserId, marks
the old key superseded with a bounded grace window during which BOTH keys authenticate, and audits the
event. `resolveByPrefix` becomes grace-aware so a superseded key works until `graceExpiresAt` then
hard-stops. No new crypto — rotation reuses the HMAC key service. This is the zero-downtime key-cycling
primitive the Developer page's rotation flow (99-07) drives.

Output: the `rotate` mutation + grace-aware resolution + `API_KEY_ROTATE` audit + the RED rotation test
GREEN.
</objective>

<threat_model>
## Trust Boundaries
| Boundary | Description |
|----------|-------------|
| rotate request → key supersession | Only an org admin (ENTERPRISE) may rotate; the old key must stop after grace |
| superseded key → auth | A superseded key must be usable only within its grace window, never after |

## STRIDE Threat Register
| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-99-05-01 | Spoofing | Superseded key usable forever | mitigate | Grace-aware `resolveByPrefix` hard-stops past `graceExpiresAt`; test both sides |
| T-99-05-02 | Elevation of Privilege | Rotation escalates scopes | mitigate | New key inherits the OLD key's scopes/actingUserId verbatim (no widening) |
| T-99-05-03 | Repudiation | Rotation not audited | mitigate | `API_KEY_ROTATE` writeAuditLog (old+new prefix + grace); lint-enforced |
| T-99-05-04 | Tampering | Unbounded grace window | mitigate | Clamp graceHours to a bounded max (168h); default 24h |
| T-99-05-SC | Tampering | npm installs | accept | No new deps |
</threat_model>
