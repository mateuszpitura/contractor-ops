### 6. Add server-side consent validation in completeStep

Modify the `completeStep` callback to validate consent server-side before marking the privacy-consent step as complete. Wrap the existing logic:

```typescript
const completeStep = useCallback(
  (stepId: string) => {
    if (completedSteps.includes(stepId)) return;

    // Server-side consent validation for privacy-consent step
    if (stepId === "privacy-consent" && !hasConsents) {
      return; // Block completion until server confirms consents
    }

    const newSteps = [...completedSteps, stepId];
    updateMutation.mutate({
      onboardingCompletedSteps: newSteps,
    });
  },
  [completedSteps, updateMutation, hasConsents],
);
```

This ensures the privacy-consent step cannot be marked complete via metadata unless `hasRequiredConsents` returns true from the server. The `OnboardingConsentStep` component calls `onComplete` (which calls `completeStep("privacy-consent")`) only after a successful `bulkGrant` mutation, but this server-side check provides defense-in-depth per D-03.

### 7. Invalidate consent query after completion

After the consent step is completed via `OnboardingConsentStep.onComplete`, the `hasRequiredConsents` query should be re-fetched. Add invalidation to the `completeStep` function or rely on the fact that `OnboardingConsentStep` already calls `bulkGrant` which triggers `onComplete`. The `hasConsents` query will be stale briefly. To fix, add a query invalidation after the consent step completes:

In the `onComplete` callback passed to `OnboardingConsentStep`, invalidate the consent query before calling completeStep:

```typescript
onComplete={() => {
  void queryClient.invalidateQueries({
    queryKey: trpc.consent.hasRequiredConsents.queryOptions().queryKey,
  });
  // Small delay to let the query refetch before gating check
  setTimeout(() => completeStep("privacy-consent"), 100);
}}
```

Actually, a cleaner approach: skip the hasConsents gate when the onComplete callback fires from OnboardingConsentStep, since that component only calls onComplete after a successful bulkGrant mutation (which means consents were just granted). Modify the completeStep to accept an optional `skipValidation` parameter:

```typescript
const completeStep = useCallback(
  (stepId: string, skipValidation = false) => {
    if (completedSteps.includes(stepId)) return;

    // Server-side consent validation for privacy-consent step
    if (stepId === "privacy-consent" && !skipValidation && !hasConsents) {
      return;
    }

    const newSteps = [...completedSteps, stepId];
    updateMutation.mutate({
      onboardingCompletedSteps: newSteps,
    });
  },
  [completedSteps, updateMutation, hasConsents],
);
```

Then pass `true` from the OnboardingConsentStep onComplete:
```typescript
onComplete={() => completeStep("privacy-consent", true)}
```

This is safe because OnboardingConsentStep only fires onComplete after a successful server-side bulkGrant mutation. The `hasConsents` gate catches any other code path that might try to skip the consent (e.g., directly calling completeStep from elsewhere).
