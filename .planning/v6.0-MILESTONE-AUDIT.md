# v6.0 Milestone Audit — Platform Maturity & Operational Hardening

**Milestone:** v6.0 (phases 70–80) · **Audited:** 2026-06-06 · **Status: ⚠ gaps_found**

Eleven phases, all marked complete. All **54/54 requirements are claimed complete in SUMMARY
frontmatter** (the REQUIREMENTS.md traceability checkboxes for FOUND6/COMPL-02·08·09·10/OFFB are
stale — superseded by the SUMMARY source-of-truth). Per-phase verification + the Phase-80 hardening
retrospective both landed a "complete pending post-deploy" verdict.

**This milestone audit found two cross-phase integration blockers that per-phase verification could
not see** — both are E2E flow breaks where a phase's own deliverables are correct in isolation but
the seam to a downstream phase was never wired. Both are **source-confirmed**, not doc-derived.

---

## 1. Blockers (must close before milestone complete)

### INT-01 · IdP deprovisioning has no UI trigger — entire F2 feature unreachable `[BLOCKER]`

`deprovisioning.startDeprovisioningRun` (`packages/api/src/routers/integrations/deprovisioning.ts:147`)
is fully implemented, cooldown-gated, signoff-gated, and QStash-fanned-out. **It has zero callers in
`apps/web-vite`** — grep finds only the mutation definition and doc comments. The offboarding
`ACCESS_REVOKE` task type exists in web-vite **only as an icon** (`task-card-run.tsx:76`) and a
template-builder option (`task-card.tsx:148`); no component wires an action to start a run.

The phase-78 web-vite UI (toggle table + run-view) landed, but the *trigger* seam from the
offboarding workflow to the saga was never connected. Result: an admin can view runs and toggle
providers but **can never start a deprovisioning** — phases 76/77/78 (15 IDP requirements, the F2
"differentiator") are dark from the UI.

> Secondary: `PROVIDERS_FOR_RUN = ['GOOGLE_WORKSPACE']` (`deprovisioning.ts:68`) — even if triggered,
> only GWS creates steps. Slack/Entra/Okta/GitHub toggles produce zero steps; the step runner
> fails-closed. (Verify the Slack leg precisely during gap closure — Phase 77 claimed it complete.)

**Affected:** IDP-01..10, IDP-12, IDP-13, IDP-15 (IDP-11 cooldown + IDP-14 scopes unaffected).

### INT-02 · Compliance hold + payment-block not released on admin approval `[BLOCKER]`

`compliance-admin.approveUploadReplacement` (`compliance-admin.ts:236`) flips the
`ContractorComplianceItem` to `status:'SATISFIED'` (`:287`) inside a `$transaction` but **never calls
`onComplianceItemSatisfied`**. That recovery hook (`compliance-recovery.ts:43`) — which re-asserts
eligibility and releases `PENDING_COMPLIANCE` approval holds — is wired into **exactly one** path:
`classification.ts:101` (classification reconcile). It is not even imported in `compliance-admin.ts`.

Result: the E2E **"portal upload → admin approve → payment unblocked"** breaks at the final step. A
contractor stays payment-blocked after their replacement document is approved, until an unrelated
classification-reconcile event or manual action fires recovery.

**Affected:** COMPL-07, COMPL-08, COMPL-11.
