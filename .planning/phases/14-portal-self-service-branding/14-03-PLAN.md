**A. Update `apps/web/src/app/[locale]/(portal)/layout.tsx`:**

Per D-12 and Research Pattern 3. Modify the existing org query to also select `settingsJson`:

```typescript
const organization = await prisma.organization.findUnique({
  where: { id: session.organizationId },
  select: { name: true, logo: true, settingsJson: true },
});

const settings = (organization?.settingsJson as Record<string, unknown>) ?? {};
const brandColor = (settings.brandColor as string) ?? null;
```

On the authenticated layout wrapper div, add style prop for CSS custom property injection:
```tsx
<div
  className="min-h-screen bg-background"
  style={brandColor ? { '--brand-accent': brandColor } as React.CSSProperties : undefined}
>
```

This injects `--brand-accent` as a CSS custom property available to all child components. Portal components can reference it via Tailwind: `text-[var(--brand-accent)]`, `bg-[var(--brand-accent)]`, `border-[var(--brand-accent)]`.

Also apply to the unauthenticated wrapper (login pages) — no, login pages don't have org context. Only apply to authenticated wrapper.

Update the PortalTopBar nav link active indicator to use brand accent when available:
In portal-top-bar.tsx, the active state uses `border-primary`. To support brand color, the active border could use `border-[var(--brand-accent,hsl(var(--primary)))]`. However, this is a portal-top-bar.tsx change — if the CSS custom property is set, we can use it in the top bar. For simplicity, the existing `border-primary` can be overridden by adding this to the layout div:
```tsx
style={brandColor ? {
  '--brand-accent': brandColor,
  '--primary': brandColor,
} as React.CSSProperties : undefined}
```

Wait — overriding `--primary` would affect ALL shadcn components. Instead, keep `--brand-accent` separate and only apply it explicitly where needed. The portal top bar active border is fine with `--primary`. The brand accent is primarily for buttons and links per D-09. This is adequate for MVP — the accent color is visible in the top bar org name area and any primary-colored buttons.

Final approach: inject only `--brand-accent`. Do NOT override `--primary` globally. Components that want brand color can use `var(--brand-accent, hsl(var(--primary)))`.

**A. Create `apps/web/src/components/settings/change-request-diff-card.tsx`:**

Per D-03 and UI-SPEC.

"use client" component.

Props:
```typescript
interface ChangeRequestDiffCardProps {
  request: {
    id: string;
    contractorName: string;
    contractorEmail: string;
    requestedChanges: Record<string, unknown>;
    previousValues: Record<string, unknown>;
    createdAt: Date | string;
    status: "PENDING" | "APPROVED" | "REJECTED";
  };
  onApproved?: () => void;
  onRejected?: () => void;
}
```

Card layout per UI-SPEC:
- CardHeader: "Profile Change Request" at `text-sm font-semibold` + contractor name + email at `text-sm text-muted-foreground` + relative timestamp (use `formatDistanceToNow` from date-fns or manual)
- CardContent: diff table with 3 columns: Field, Current Value, Requested Value
  - Map over keys of requestedChanges. For each key:
    - Field column: human-readable label (bankAccountNumber -> "Bank Account", bankName -> "Bank Name", swiftBic -> "SWIFT/BIC", taxId -> "Tax ID")
    - Current Value column: `previousValues[key]` at `text-sm`
    - Requested Value column: `requestedChanges[key]` at `text-sm font-semibold`
    - Changed rows: `bg-muted` background
  - Use a simple HTML table with proper th/td for accessibility
- Action row (only for PENDING status): "Approve Changes" primary Button + "Reject Changes" outline destructive Button

Approve action: call `trpc.settings.reviewChangeRequest.mutate({ requestId: request.id, action: "approve" })`. Success toast: "Change request approved. Contractor profile updated." Call onApproved callback.

Reject action: open Dialog per UI-SPEC:
- Title: "Reject Change Request"
- Body: "Optionally provide a reason for rejection. The contractor will be notified."
- Textarea with placeholder "Reason for rejection (optional)"
- "Confirm Rejection" destructive Button
- On confirm: call `trpc.settings.reviewChangeRequest.mutate({ requestId: request.id, action: "reject", comment })`. Success toast: "Change request rejected. Contractor has been notified." Close dialog. Call onRejected callback.

For APPROVED/REJECTED status: show Badge with status (per Status Badge Mapping in UI-SPEC) instead of action buttons.
