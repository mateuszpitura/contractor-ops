**Part B: Wire Jira webhook processing in the _process route and JiraAdapter (per D-07).**

Do NOT create a separate `/api/webhooks/jira/route.ts`. Jira webhooks MUST flow through the existing unified pipeline:
1. `/api/webhooks/jira` (the `[provider]` catch-all route) receives the webhook
2. Calls `JiraAdapter.verifyWebhookSignature` — must return `{ valid, eventType, organizationId?, connectionId? }`
3. Creates WebhookDelivery
4. Enqueues to QStash with `{ deliveryId, provider: "jira" }`
5. `_process` route receives from QStash, dispatches to Jira handler

**CRITICAL: Do NOT import `@contractor-ops/api` from `packages/integrations`.** This would create a circular monorepo dependency (`packages/api` already imports from `packages/integrations`). Instead, keep `JiraAdapter.handleWebhook` as a no-op/passthrough and dispatch the actual processing in `apps/web` where both packages are safely importable.

**JiraAdapter changes (jira-adapter.ts):**
- Set `supportsWebhooks = true`
- Implement `verifyWebhookSignature` to validate the Jira webhook payload and return `{ valid, eventType, organizationId, connectionId }`. Since Jira dynamic webhooks are registered per-connection, the webhook URL registered with Jira should include `connectionId` as a query param. The unified [provider] route passes all headers and the request URL to `verifyWebhookSignature`. Two approaches (choose whichever aligns with the existing adapter pattern):
  - Embed connectionId in a query param during registration and extract from the request URL
  - OR look up the connection by matching the webhook secret/payload project to the registered connection
- The `verifyWebhookSignature` return type must include `connectionId` and `organizationId` to match how the unified route uses them when creating the WebhookDelivery record.
- Implement `handleWebhook` as a **no-op** that simply returns. The actual processing is dispatched from the `_process` route (see below). Add a comment: `// Jira webhook processing dispatched from _process route to avoid circular package dependency`

**_process route changes (apps/web/src/app/api/webhooks/_process/route.ts):**
Add a Jira-specific dispatch block AFTER the `adapter.handleWebhook` call, following the same pattern as the existing e-sign provider block. The `_process` route lives in `apps/web` which can safely import from both `@contractor-ops/api` and `@contractor-ops/integrations` without circular dependencies:

```typescript
// After adapter.handleWebhook(delivery.payloadJson, ...) call:

// For Jira provider, dispatch to processJiraWebhook from @contractor-ops/api

if (provider === "jira") {
```

Place this block after the `adapter.handleWebhook` call and before the e-sign provider block (or alongside it — both are provider-specific post-processing). This mirrors exactly how the e-sign completion handling already works in this file: the adapter call runs first (no-op for Jira), then provider-specific logic in the route dispatches to the appropriate service.

**Part C: Inject outbound Jira transition into workflow task status updates (per D-06, JIRA-03).**

In `packages/api/src/routers/workflow.ts`, locate `completeTask` (~line 1008) and `skipTask` (~line 1080) mutations. After the task status is updated and the transaction commits, add a post-transaction hook to trigger outbound Jira sync:

```typescript
// After the $transaction block in completeTask:
// Fire-and-forget outbound Jira transition (non-blocking)
if (result.externalRefType === "JIRA_ISSUE" && result.externalRefId) {
  void (async () => {
    try {
      const { transitionJiraIssue } = await import("../services/jira-issue-sync.js");
      const connection = await prisma.integrationConnection.findFirst({
        where: { organizationId: ctx.organizationId, provider: "JIRA", status: "CONNECTED" },
        select: { id: true },
      });
      if (connection) {
        await transitionJiraIssue(prisma, ctx.organizationId, connection.id, result.id, "DONE");
      }
    } catch (err) {
      console.error("[workflow/completeTask] Outbound Jira transition failed:", err);
    }
  })();
}
```

Apply the same pattern to `skipTask` with status "SKIPPED".

IMPORTANT: The outbound sync MUST be fire-and-forget (non-blocking). Do not let a Jira API failure prevent the workflow task from completing. The `result` variable from the `$transaction` already has `externalRefType` and `externalRefId` — ensure the `include` in the task query includes these fields.

Ensure the task findFirst query inside each transaction includes `externalRefType` and `externalRefId` in its select/return so they are available after the transaction.
