Boundary verification (mental check before writing):

- `endDateMs == overdueStartMs` (i.e., dueDate + exactly 1 day): `floor(0 / day) + 1 = 1` ✓ (one full overdue day elapsed: overdueStartMs itself)
- `endDateMs == overdueStartMs + 1 day - 1ms`: `floor((day-1ms)/day) + 1 = 0 + 1 = 1` ✓ (still day 1, partial day rounds down)
- `endDateMs == overdueStartMs + 1 day`: `floor(day/day) + 1 = 1 + 1 = 2` ✓
- Feb-13-due, Mar-15-asOf (existing test fixture, 30 elapsed days): overdueStartMs = Feb 14. (Mar 15 - Feb 14) = 29 days → `29 + 1 = 30`? **No** — that would keep the test passing with the broken formula's value.

**Re-derivation:** The CONTEXT.md / RESEARCH claim is that the test expectation `daysOverdue=30` is itself the bug. The LPCDA-correct interpretation: due Feb 13 → grace day Feb 13 → first overdue day Feb 14 → ... → as of Mar 15, the buyer has been overdue for the days Feb 14, 15, 16, ..., Mar 14 (last full overdue day before asOf). That's **29 days**, not 30. So the formula must be `floor((endDateMs - overdueStartMs) / day)` (no `+ 1`).

**Boundary re-check with the no-+1 formula:**
- `endDateMs == overdueStartMs` (Feb 14 00:00:00, asOf): `floor(0) = 0` — buyer has been "in the overdue state" since this very moment but no full day has elapsed yet → 0 is defensible (matches the existing line-208 test that asserts `daysOverdue=0` "asOf exactly on due date" semantics).
- `endDateMs == overdueStartMs + 1 day`: `floor(1) = 1` — Feb 14 has fully elapsed → 1 ✓
- Feb-13-due, Mar-15-asOf: (Mar 15 - Feb 14) = 29 → `floor(29) = 29` ✓ matches CONTEXT.md / RESEARCH expected value.

**Therefore the correct edit is WITHOUT the `+ 1`:**

```ts
  // LPCDA Section 4(1): interest accrues from the day AFTER due date
  // (overdueStartMs). daysOverdue counts the number of full days that
  // have ELAPSED since overdueStartMs — partial days don't accrue a full
  // day's interest. Fixes B-05 (Phase 65 CONTEXT.md decision D-03 /
  // .planning/phases/63-uk-payments-financial-features/63-VERIFICATION.md).
  // DO NOT revert to (endDateMs - dueDateMs) — that overstates by one day
  // because it counts the grace-period day (= dueDate itself) as overdue.
  const daysOverdue = Math.floor((endDateMs - overdueStartMs) / (24 * 60 * 60 * 1000));
```

Apply that edit (no `+ 1`).

**Sanity check the existing line-199 early-return still does the right thing:** the guard `if (endDateMs <= dueDateMs)` returns `daysOverdue: 0` before our new formula runs. If `dueDateMs < endDateMs <= overdueStartMs - 1ms` (i.e., the same calendar day as `dueDateMs` but later in the day, when we model dates as midnight UTC this branch doesn't really exist for `@db.Date` columns), the new formula yields `floor(negative-or-near-zero / day) = -1 or 0`. To eliminate any risk of a negative result, also widen the early-return to `endDateMs < overdueStartMs`:

Change line 199 from:

```ts
  if (endDateMs <= dueDateMs) {
```

to:

```ts
  // Cover the entire grace period (up to but not including overdueStartMs)
  // so the post-guard daysOverdue formula can never return a negative value
  // when endDateMs lands inside the same calendar day as dueDateMs.
  if (endDateMs < overdueStartMs) {
```

This is a safe widening — for `@db.Date` columns midnight-UTC values, `endDateMs == dueDateMs` (early-return triggered before, still triggered now) and `endDateMs == overdueStartMs` (= dueDateMs + 24h) does NOT trigger early-return and falls through to the formula returning `0`. Both paths produce `daysOverdue: 0` for the boundary, consistent with the existing test at line 208 ("asOf exactly on due date").
