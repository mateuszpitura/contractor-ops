    ```typescript
    testCourierConnection: adminProcedure
      .input(z.union([dpdConfigSchema, upsConfigSchema]))
      .mutation(async ({ input }) => {
        const { carrier, ...credentials } = input;
        try {
          const client = getCourierClient(carrier, credentials);
          // Attempt a lightweight API call to verify credentials work.
          // getStatus with a dummy ID will authenticate and return an error
          // about the shipment not existing (not an auth error).
          // If auth fails, it throws before reaching the API response.
          await client.getStatus("TEST_CONNECTION_PROBE");
          return { success: true };
        } catch (error) {
          // If the error is about the shipment not being found, that means
          // auth succeeded -- the API accepted our credentials.
          const msg = error instanceof Error ? error.message : String(error);
          if (msg.includes("not found") || msg.includes("NOT_FOUND") || msg.includes("404")) {
            return { success: true };
          }
          return { success: false, error: "Connection failed. Check your credentials." };
        }
      }),
    ```

    Key details:
    - Uses `adminProcedure` (same as `saveCourierConfig`) -- only org admins can test connections
    - Accepts same input schema as `saveCourierConfig` (`z.union([dpdConfigSchema, upsConfigSchema])`) -- this matches the client-side call pattern where the full credential object with carrier field is passed
    - Uses `getCourierClient` from carrier-factory.ts to instantiate the correct client
    - Calls `client.getStatus("TEST_CONNECTION_PROBE")` as a lightweight auth probe
    - Returns `{ success: true }` on successful auth (including "shipment not found" errors which prove auth worked)
    - Returns `{ success: false, error: string }` on auth failure -- never exposes raw error details to client
    - Does NOT throw TRPCError on connection failure -- returns structured result so the UI can show toast
