## Phase Nature

This phase is a **mechanical cleanup** with no new UI components, screens, or interactions. It performs two tasks:

1. Wraps 3 existing provider section components with the established `FeatureGate` component (Phase 35 pattern).
2. Rebuilds API dist types and removes 13 `(trpc as any)` proxy workarounds.

The design contract below specifies the FeatureGate integration points and the upgrade banner copy. All spacing, typography, color, and component patterns are inherited from the existing design system without modification.

## FeatureGate Integration Points

These are the 3 components that receive the `FeatureGate` wrapper in this phase.

| Component | File | `requiredTier` | `featureName` |
|-----------|------|---------------|---------------|
| `JiraProviderSection` | `components/integrations/jira-provider-section.tsx` | `"Pro"` | `"Jira integration"` |
| `OrgCalendarSection` | `components/settings/org-calendar-section.tsx` | `"Pro"` | `"Calendar integration"` |
| `MyCalendarSection` | `components/settings/my-calendar-section.tsx` | `"Pro"` | `"Calendar integration"` |

**Wrapping pattern:** Wrap the outermost `return` JSX with `<FeatureGate>`, identical to the pattern in `linear-provider-section.tsx`, `gws-provider-section.tsx`, and `teams-provider-section.tsx`.

**Loading behavior:** `FeatureGate` renders children while loading subscription data (Phase 35 decision). Calendar sections render their own skeleton loading for connection data. No double-skeleton issue.
