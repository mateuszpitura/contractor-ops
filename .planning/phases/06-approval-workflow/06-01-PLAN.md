    **3. Create `packages/api/src/services/approval-engine.ts`:**

    Import Prisma types from `@contractor-ops/db`. Define helper `addHours(date: Date, hours: number): Date` inline (same pattern as Phase 4 decision [04-01]).

    **evaluateConditions function:**
    - Takes `conditionsJson: unknown` and `invoice: { totalGrosze: number; contractorType?: string }`
    - Parse conditionsJson as array, return false if null/empty
    - Every condition must match (AND logic): `amount` field compares `invoice.totalGrosze` against `value * 100` (grosze conversion), `contractorType` field compares with string equality
    - Return boolean

    **routeToChain function:**
    - Takes `tx` (Prisma transaction client), `organizationId`, `invoice` data
    - Queries all active INVOICE chain configs ordered by createdAt ASC
    - First-match: iterate chains, call evaluateConditions, return first match
    - Fallback: return chain where isDefault=true, or null if none

    **createApprovalFlow function:**
    - Takes tx, organizationId, resourceType ("INVOICE"), resourceId, chainConfig, createdByUserId
    - Deep-clone stepsJson from chainConfig
    - Resolve role-based approvers: for each step with approverRole but no approverUserId, query the first user with that role in the organization. If no user found, throw TRPCError PRECONDITION_FAILED "No user with role {role} found"
    - Create ApprovalFlow with status PENDING, currentStepOrder 1
    - Create ApprovalStep rows: first step gets status PENDING and slaDeadline = addHours(now, step.slaHours), rest get NOT_STARTED and null slaDeadline
    - Return flow with steps included

    **advanceFlow function:**
    - Takes tx, flowId
    - Load flow with all steps ordered by stepOrder
    - Find next step (status NOT_STARTED, stepOrder > currentStepOrder)
    - If next step exists: set it to PENDING, set slaDeadline = addHours(now, slaHours from stepsJson), update flow.currentStepOrder
    - If no next step: mark flow APPROVED, set completedAt = now, return { completed: true, flowStatus: "APPROVED" }
    - Return { completed: false, nextStepOrder }

    **computeSlaStatus function:**
    - Takes slaDeadline (Date | null), status (string)
    - If no deadline or status !== "PENDING", return null
    - Calculate remainingMs = deadline - now
    - If remainingMs <= 0: return { status: "overdue", label: `OVERDUE ${hours}h` }
    - Else compute percentage: slaHours is inferred from (slaDeadline - step createdAt). Use remainingMs / totalMs ratio.
    - >50% = "green", 25-50% = "yellow", <25% = "red"
    - Return { status, remainingMs, label: `${hours}h left` }

    Note: Store slaHours on step config so percentage can be computed. The step's slaDeadline minus slaHours*3600000 gives the activation time.

    **Approval Queue:**

    `listPending`: tenantProcedure.use(requirePermission({ invoice: ["approve"] })).input(approvalQueueSchema).query -- Query ApprovalStep with:
    - where: organizationId, status filtering (if "pending" -> PENDING, if "overdue" -> PENDING + slaDeadline < now)
    - If tab="my": add approverUserId = ctx.userId
    - Include: approvalFlow (include: { steps: false }), get resourceId from flow to fetch Invoice data (invoiceNumber, sellerName, totalGrosze, currency, createdAt)
    - Use raw query or nested include to get invoice data. Simplest: include approvalFlow, then batch-fetch invoices by resourceId.
    - Sort: slaDeadline ASC (overdue first naturally). For "overdue" status filter: slaDeadline < new Date() AND status = "PENDING".
    - Paginate: skip = (page - 1) * pageSize, take = pageSize
    - Count total for pagination
    - For each step, compute SLA status using computeSlaStatus
    - Return { items: plain(enrichedSteps), total, page, pageSize }

    **Approval Actions (all wrapped in prisma.$transaction):**

    `approve`: tenantProcedure.use(requirePermission({ invoice: ["approve"] })).input(approveStepSchema).mutation:
    1. In $transaction: find step by id, verify organizationId matches, verify status is PENDING, verify approverUserId = ctx.userId
    2. Create ApprovalDecision: { organizationId, approvalStepId: step.id, actorUserId: ctx.userId, decision: "APPROVE", comment: input.comment }
    3. Update step: status = "APPROVED", actedAt = now, decision = "APPROVE"
    4. Call advanceFlow(tx, step.approvalFlowId)
    5. If flow completed (advanceFlow returns completed=true): update Invoice status to "APPROVED" via tx.invoice.update where id = flow.resourceId
    6. Return plain(updatedStep)

    `reject`: Same pattern but:
    - Decision: "REJECT", comment is required (min 10 chars from schema)
    - Step: status = "REJECTED"
    - Flow: status = "REJECTED", completedAt = now
    - Invoice: status = "REJECTED"
    - Do NOT advance to next step

    `delegate`: tenantProcedure.use(requirePermission({ invoice: ["approve"] })).input(delegateStepSchema).mutation:
    1. Verify step is PENDING and assigned to ctx.userId
    2. Verify delegateToUserId exists in the organization
    3. Create ApprovalDecision with decision "DELEGATE"
    4. Update step: approverUserId = input.delegateToUserId
    5. Step remains PENDING (SLA continues per D-10)

    `requestClarification`: tenantProcedure.use(requirePermission({ invoice: ["approve"] })).input(requestClarificationSchema).mutation:
    1. Verify step is PENDING and assigned to ctx.userId
    2. Create ApprovalDecision with decision "REQUEST_CHANGES", comment
    3. Step remains PENDING (approver still needs to decide per research recommendation)

    **Bulk Actions:**

    `bulkApprove`: tenantProcedure.use(requirePermission({ invoice: ["approve"] })).input(bulkApproveSchema).mutation:
    - Use Promise.allSettled to process each stepId individually
    - Each item: run the same approve logic in its own $transaction
    - Return { succeeded: number, failed: number, errors: string[] }

    `bulkReject`: Same pattern with shared comment from input applied to all.

    **Submit for Approval:**

    `submitForApproval`: tenantProcedure.use(requirePermission({ invoice: ["update"] })).input(z.object({ invoiceId: z.string() })).mutation:
    1. In $transaction: find invoice, verify org, verify status allows submission (matchStatus in ["MATCHED", "MANUALLY_CONFIRMED"] and status not already "APPROVAL_PENDING")
    2. Call routeToChain(tx, organizationId, { totalGrosze: invoice.totalGrosze }) to find matching chain
    3. If no chain found, throw BAD_REQUEST "No approval chain configured for this organization"
    4. Call createApprovalFlow(tx, { organizationId, resourceType: "INVOICE", resourceId: invoice.id, chainConfig, createdByUserId: ctx.userId })
    5. Update invoice: status = "APPROVAL_PENDING"
    6. Return plain(flow)

    **Audit Trail:**

    `getAuditTrail`: tenantProcedure.use(requirePermission({ invoice: ["read"] })).input(z.object({ invoiceId: z.string() })).query:
    - Find ApprovalFlow where resourceId = invoiceId, organizationId
    - If no flow, return { events: [] }
    - Find all ApprovalDecisions through flow's steps, include actor (User: id, name, email, image)
    - Build timeline: combine human decisions with derived system events:
      - System event: "submitted" from flow.startedAt
      - System event: "routed" from flow.chainConfigId (fetch chain name)
      - System events: SLA breaches computed from steps where slaDeadline < actedAt or slaDeadline < now and status still PENDING. **Canonical SLA breach event shape:** `{ type: "system", label: "sla_breached", levelName: step.name, timestamp: step.slaDeadline.toISOString() }`. This is the authoritative definition -- Plan 05 verifies this exists but does not re-implement it.
    - Sort all events by createdAt DESC (most recent first per D-11)
    - Return plain({ events })
