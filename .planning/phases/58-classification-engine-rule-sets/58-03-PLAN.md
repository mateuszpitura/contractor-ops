<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → tRPC | Untrusted answer payload (could be arbitrary types) — Zod at entry |
| router → engine | Server invokes pure scoreAssessment; engine never accepts client-computed outcome |
| org A ctx → org B row | tenantProcedure + Prisma extension enforce multi-tenant scope; test must prove leakage is blocked |

## STRIDE Threat Register (applicable ASVS: V2, V4, V5, V7, V8, V13)

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-58-09 | Information Disclosure | classification router reads | mitigate | `tenantProcedure` (auth + org scope). Integration test: Org A caller cannot read Org B's assessmentId (returns NOT_FOUND, not 403 — V7 leakage-resistant error). |
| T-58-10 | Tampering | saveAnswer | mitigate | (V5) Zod schema per questionId's answerType; reject writes when status !== 'draft'; idempotent upsert so replay is safe. |
| T-58-11 | Tampering | client-computed outcome | mitigate | `submit` computes outcome server-side only; `outcome` input field rejected by schema; scoreAssessment lives in a server-only import (Plan 02 profile class). |
| T-58-12 | Repudiation | submit | mitigate | `immutableAfter` timestamp set atomically with status='completed'; append-only — re-submit throws CONFLICT. |
| T-58-13 | Denial of Service | saveAnswer autosave loop | mitigate | (V13) Upstash ratelimit 120/min per assessmentId; exceeding returns TOO_MANY_REQUESTS. |
| T-58-14 | Repudiation | acknowledgeDisclaimer bypass | mitigate | Only completed rows accepted; timestamp is append-only (idempotent overwrite only of the acknowledgement field, not dismissal). |
| T-58-15 | Info Disclosure | PII in answers → logs | mitigate | (V8) Observability middleware excludes request body for classification.* procedures; audit writes only metadata (assessmentId, action, userId, timestamp). |
| T-58-16 | Integrity | rule-set drift on resume | mitigate | `getDraft` compares draft.ruleSetVersion vs current profile.ruleSetVersion → throws PRECONDITION_FAILED if stale (Pitfall 7). |
| T-58-17 | Tampering | concurrent two-tab autosave | mitigate | saveAnswer accepts optional `expectedUpdatedAt`; if row.updatedAt > expectedUpdatedAt, throw CONFLICT. Client reloads latest + shows toast. |
</threat_model>

2. Create `packages/api/src/middleware/classification-rate-limit.ts`:
   ```typescript
   import { TRPCError } from '@trpc/server';
   import { Ratelimit } from '@upstash/ratelimit';
   import { Redis } from '@upstash/redis';
   import { t } from '../init.js'; // or wherever middleware is exported from

   const redis = Redis.fromEnv();
   const saveAnswerLimiter = new Ratelimit({
     redis,
     limiter: Ratelimit.slidingWindow(120, '1 m'),
     prefix: 'classification:saveAnswer',
     analytics: false,
   });

   export const classificationSaveAnswerRateLimit = t.middleware(async ({ ctx, input, next }) => {
     const assessmentId = (input as { assessmentId?: string })?.assessmentId;
     if (!assessmentId) throw new TRPCError({ code: 'BAD_REQUEST' });
     const key = `${ctx.organizationId}:${assessmentId}`;
     const { success, remaining } = await saveAnswerLimiter.limit(key);
     if (!success) {
       throw new TRPCError({
         code: 'TOO_MANY_REQUESTS',
         message: 'Autosave rate limit exceeded (120/min per assessment). Slow down.',
       });
     }
     return next({ ctx });
   });
   ```
   If existing middleware pattern constructs Ratelimit differently, follow the repo pattern exactly (same prefix style, same Redis.fromEnv / in-memory fallback).
