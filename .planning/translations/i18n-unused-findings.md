## False-positive controls applied

The script is reverse-direction of `scripts/audit-i18n-code-coverage.ts` and
adds three classes of coverage to avoid flagging live keys:

1. **Wildcard namespaces** — when a bound `t` is hit with a non-literal arg
   (e.g. `useTranslatedError` does `t(errorKey)` where `errorKey` comes from
   the tRPC error shape), the whole namespace is marked as reachable.
   38 namespaces today: `Errors`, `Common`, `Navigation`, `Approvals`,
   `Contractors`, `Contracts`, `Dashboard`, `Invoices`, `Payments`,
   `Organization`, `Search`, `Settings.tabs`, …
2. **Floating literals** — when `const { t } = useXxxTab()` destructures
   the bound translator from a custom hook, the caller's `t('createDialog.title')`
   call has no local namespace. The auditor collects every such literal and
   resolves it against any top-level NS whose tree contains a matching leaf.
   Recovered 327 leaves (e.g. `Settings.apiKeys.createDialog.*`).
3. **Bare literals** — full `'NS.path.leaf'` strings anywhere in source
   (covers `COMMON_TOAST` constants, `tKey(t, 'X')`, `useResourceMutation({ successMessage: 'NS.leaf' })`).

## Concrete findings worth a follow-up

### 1. `Api.*` — 59/59 dead AND there's a latent bug

`Api.email.subject.*` keys are listed in the server's
`EMAIL_SUBJECT_KEYS` constant (`packages/api/src/services/email-templates.ts:14`)
in the form `'email.subject.approvalRequest'` — **without** the `Api.` prefix.
The `subject` value is then cast `String(subject)` at
`packages/api/src/services/notification-service.ts:184`, which stringifies the
`{key, params}` object as `"[object Object]"`. The keys are never resolved
through `i18next` because (a) the server is a Fastify Node app with no i18n
runtime and (b) no caller passes the `labels?:` prop into the React Email
templates, so the templates fall back to hardcoded English at
`packages/api/src/emails/approval-request.tsx:32-37`.

→ Action: either wire a server-side i18n resolver and pass `labels` to each
template, or delete the whole `Api.*` namespace from the four locale files
(and the EMAIL_SUBJECT_KEYS dead code) and accept English-only subject lines.
Either choice removes 59 dead leaves × 4 locales.

### 2. `Notifications.center.*` — duplicate of top-level `Notifications.*`

`Notifications.center.{pageTitle, markAllRead, unreadOnly, filterAll, …}`
mirrors the actually-used `Notifications.{title, markAllRead, unreadOnly, …}`
(see `apps/web-vite/src/components/notifications/notification-center-container.tsx:32-114`).
The `center` subtree was likely a parallel scaffold that lost the race.

→ Action: delete `Notifications.center.*`, `Notifications.reminderEditor.*`,
`Notifications.preferences.*`, `Notifications.integrations.*`,
`Notifications.userMapping.*` — confirmed no source references in the
monorepo. Net: 156 leaves × 4 locales.

### 3. `Teams.cards.*` — Microsoft Teams adaptive-card copy never rendered

`packages/api/src/services/messaging/teams-messaging-provider.ts:176`
implements `sendChannelAlert` but does not read any of `Teams.cards.{approve,
reject, approvalTitle, rejectModalTitle, …}` — the adaptive card body is
likely built inline / from another source.

→ Action: confirm with whoever owns the Teams integration whether these are
intended for an upcoming card refresh; otherwise delete (12 leaves × 4).

### 4. `Workflow.*` (singular, 5 leaves, 100 % dead)

Likely a one-off typo of `Workflows.*` — entire top-level NS is dead and
has only 5 keys. Verify nothing dynamic resolves them; delete.

### 5. Pre-built KSA / EU compliance UI (`Zatca.*`, `TaxAdmin.*`, `EInvoice.*`)

These namespaces look pre-localized for features that haven't shipped yet
(82–91 % dead per namespace). Per `[[project_local_only_legal_deferred]]`
the app is local-only with legal sign-off deferred — none of these screens
are wired to user-facing routes.

→ Action: **don't** delete blindly. Mark as "scaffold-pending" and re-run
the audit after each feature lands. Worth adding a `// TODO(unleash:zatca-csid)`
or similar pointer for the ramp.

### 6. `Settings.integrations.*` (117 dead) and `Settings.auditLog.*` (106 dead)

The Settings shell uses ~285 live keys out of 740. The two biggest dead
subtrees are integrations + auditLog. Spot-check before bulk delete: many
sub-tabs hide behind feature flags or run-time `t(integrationId)` patterns;
ensure those aren't already covered by `Settings.tabs` (wildcard).

## Operational integration

Add to `package.json` scripts (suggestion):

```json
"i18n:dead": "tsx scripts/audit-i18n-unused-keys.ts",
"i18n:dead:dump": "tsx scripts/audit-i18n-unused-keys.ts --dump"
```

Run with `--dump` to refresh `.planning/translations/i18n-unused-en.json`
(map of `unused-key → EN value`) + `.planning/translations/i18n-unused-report.json`
(machine-readable summary).

Exit code is always 0 — this is a hygiene gauge, not a build gate. The build
gate stays on `scripts/audit-i18n-code-coverage.ts` (catches the opposite
direction: code refs missing from locales).

## Limitations / known imprecision

1. **Floating-literal resolution is over-permissive by design.** A
   `t('title')` in a file whose hook returns a `Settings.apiKeys` binding
   will mark `Settings.apiKeys.title`, `Settings.title`, `Errors.title`, etc.
   as used if any of them exist. This is intentional — we'd rather keep a
   live key than recommend deleting one. The trade-off is that some truly
   dead keys remain hidden in noisy namespaces (any NS where `title`,
   `description`, `name` appear).
2. **Wildcard NS bloat.** `Common` is a wildcard (`useCommonToasts` uses
   `t(key)` indirectly). This hides any dead keys inside `Common`. Manual
   spot-check needed if shaving `Common` is on the table.
3. **No cross-file hook-return tracing.** The auditor scans each file
   independently. If a chain of three hooks transports a binding two levels
   deep, the floating-literal pass still catches it via top-level NS
   matching, but it's not deterministic.
4. **Tests + generated dirs are skipped.** `__tests__`, `*.test.tsx`,
   `src/generated/`, `node_modules` are excluded from the source scan.
